# M5 implementation notes (backend half)

Design decisions and deviations made while implementing the backend half of M5 (resources, coaches,
capacity). Primary spec: `../../utkast-kravspec.txt` §12 (resurser och kapacitet — §12.4's example
is reproduced exactly, see below), §13 (tränare), §6.7–6.10 (TimeSlot/Court/TrainingBlock/Assignment
terminology), §7.7–7.10 (schema); design: `docs/design/02-product-data-ui.md` §4–§5. The frontend
half (Tider & banor / Tränarvyn / Kapacitetsvyn UI) is a separate task.

## Schema

**One migration, no new tables.** `venue`, `court`, `time_slot`, `training_block`, `coach_profile`,
`coach_time_slot` were all already created by `V1__core.sql` (M1), anticipating M5 — confirmed by
re-reading V1 before writing any code, same practice as M4. `V4__resources.sql` adds:

1. `CREATE UNIQUE INDEX idx_court_venue_id_name ON court (venue_id, name)` — backs the idempotent
   "Bana N" auto-creation lookup and gives a free 409 (via the existing
   `DataIntegrityViolationException` → `ApiExceptionHandler` path) on a manually-created duplicate
   court name within one venue.
2. `ALTER TABLE training_block ADD COLUMN deactivation_source TEXT` (M5 review fix 1) — `MANUAL` |
   `SHRINK` | `NULL`-while-active; see "TrainingBlock generation" below. Plain `ADD COLUMN` is
   supported natively by SQLite, so neither change needs the create-copy-rename idiom.

## Time slots (spec §12.1)

`GET|POST /api/plans/{planId}/time-slots`, `GET|PATCH|DELETE /api/time-slots/{id}` (nested-collection
/ flat-resource split mirrors the existing `ParticipantProfileController` convention).

- **Exact-duplicate detection is application-level, not a DB `UNIQUE`**: `date` is nullable (a
  recurring dayOfWeek-based slot has no date), and SQL/SQLite treat every `NULL` as distinct from
  every other `NULL`, so a `UNIQUE(activity_plan_id, day_of_week, date, start_time, end_time)` index
  would silently fail to catch two dayOfWeek-based duplicates. `TimeSlotRepository.findDuplicate`
  uses `IS` (SQLite's null-safe equality operator) instead, and the controller throws the new
  `ConflictException` (409) — same shape as the constraint-violation 409 path, added because this
  is the first case in the codebase where a duplicate needs app-level, not DB-level, detection.
  Overlap between two non-identical slots (spec: allowed, since they represent different courts) is
  simply never checked.
- **`label` auto-regeneration on PATCH**: since there's no separate "was this label manually
  customized" column, `TimeSlotController.isAutoGeneratedLabel` re-derives what the label *would
  have been* for the slot's current day/date/times and compares it to the stored value. If they
  match (the label was never touched by hand), a PATCH that changes day/date/start/end regenerates
  the label to match; if they differ (a genuine custom label like "Kvällspass"), it's preserved
  across unrelated PATCHes. Explicit `"label": null` always forces regeneration; explicit
  `"label": "..."` always sets exactly that text.
- Time format is strictly `HH:mm` (validated, then re-normalized through `LocalTime` so e.g. a
  hypothetical single-digit-hour input can't produce an inconsistent stored form); label time
  rendering uses `.` per the spec's own examples ("18.00", not "18:00") joined with an en dash "–".
- `dayOfWeek` stores the `java.time.DayOfWeek` enum name (`THURSDAY`, ...), matching V1's existing
  column comment. **Exactly one** of `dayOfWeek`/`date` must be set (M5 review fix 5): both missing
  and both present are each a clear 400 — "both" is ambiguous (which wins if the date's actual
  weekday contradicts `dayOfWeek`?). A PATCH converting a weekly slot to a dated one must clear
  `dayOfWeek` in the same request (`{"date": "...", "dayOfWeek": null}`).
- **Schedule ordering is Java-side** (M5 review fix 4): `day_of_week` stores enum *names*, whose
  lexicographic order (FRIDAY < THURSDAY < TUESDAY) is not weekday order.
  `TimeSlotRepository.findByActivityPlanId` sorts by weekday ordinal (Monday=1..Sunday=7, derived
  from `date` for dated-only slots), then concrete date, then start time, then id — and since the
  slot list, the grouped training-blocks view, and the capacity per-slot breakdown all read through
  that one method, they share the ordering.

## Courts, venues, and TrainingBlock generation (spec §12.2/§12.3)

`GET|POST /api/venues` (nested `courts`), `GET|PATCH|DELETE /api/venues/{id}`; `GET|POST
/api/courts`, `GET|PATCH|DELETE /api/courts/{id}` — both club-level (no `activityPlanId`/
`seasonPlanId`, per the existing V1 schema). `POST /api/courts` without a `venueId` (and the block
generator, below) resolve the "default venue" via `DefaultVenueService`: the first venue by id
(UUIDv7 sort ≈ creation order) if any venue exists, else auto-creates one named **"Hallen"**.

**The core endpoint**: `PUT /api/plans/{planId}/time-slots/{slotId}/courts {count: N}` →
`TrainingBlockGenerationService.generateForSlot`. Courts are club-wide, not slot-scoped: courts named
exactly `"Bana " + n` are found-or-created under the default venue for `n = 1..N` and **reused**
across every time slot that needs an n-th court (three slots declared 4/4/3 share the same physical
`Bana 1..4` pool → 4+4+3 = 11 blocks, verified in the E2E transcript below). `TrainingBlock` rows are
matched to a court by parsing the trailing integer out of its name (`^Bana (\d+)$`); blocks whose
court doesn't match that pattern (a custom-named court a user manually associated some other way —
not something this endpoint itself ever produces) are left completely untouched by regeneration, a
deliberate MVP simplification.

- **Idempotent & identity-preserving**: `UNIQUE(time_slot_id, court_id)` (already in V1) means the
  same (slot, court) pair always maps to the same row; calling with an unchanged count makes zero
  writes; shrinking never deletes (only deactivates, spec §12.3's "MVP kan hantera detta genom att
  användaren kan inaktivera en TrainingBlock"); growing back to a previous count reactivates the
  *same* row ids. Verified by `TrainingBlockGenerationServiceTest`-equivalent cases in
  `TrainingBlockControllerTest` (idempotency, shrink/grow identity, deactivate-beyond-N).
- **Manual §12.3 exceptions survive regeneration** (M5 review fix 1; replaces the first
  implementation's "count is authoritative" rule): every inactive block records *why* it is inactive
  in `training_block.deactivation_source` — `MANUAL` (set by `PATCH /api/training-blocks/{id}
  {active:false}`) or `SHRINK` (set by the generation shrink path). Re-declaring `count = N` only
  auto-reactivates blocks the generator deactivated itself (`SHRINK`, or a legacy `NULL`); a
  `MANUAL` block stays inactive until the user explicitly PATCHes `{active:true}`, which also clears
  the marker and returns the block to normal generation management. An already-`MANUAL` block whose
  court index falls outside a shrunken count keeps its `MANUAL` marker (the user's explicit
  exception outranks the generator's bookkeeping). Regression-tested in
  `TrainingBlockControllerTest.manualDeactivationSurvivesRegenerationUntilManuallyReactivated`.
- `GET /api/plans/{planId}/training-blocks` returns the grouped view (`{timeSlot, blocks[]}[]`),
  sorted by day/date/start then court index, courts beyond the `Bana N` pattern sorted last.

## Coaches (spec §13.1)

`GET|POST /api/plans/{planId}/coaches`, `GET|PATCH|DELETE /api/coaches/{id}`, plus `GET|PUT
/api/plans/{planId}/coaches/{id}/availability` and `GET|PUT
/api/plans/{planId}/coaches/{cid}/field-values` (the last one not explicitly enumerated in the M5
brief's endpoint list, but promised by `FieldValueService`'s own M4 Javadoc — "the same service backs
the coach-scoped endpoint arriving in M5" — and trivial to add by mirroring
`ParticipantFieldValueController`).

- **Create-from-person or new person**: `POST` accepts either `personId` (an existing person — their
  `canBeCoach` flag is flipped on if not already set, spec §13.2 "samma person ska kunna vara både
  tränare och deltagare") or `firstName`+`lastName` (+ optional `email`/`phone`) to create a new
  `Person` with `canBeCoach=true`, `canBeParticipant=false`. `UNIQUE(person_id, activity_plan_id)`
  (already in V1) gives the "one coach profile per person per plan" 409 for free.
- **Level validation**: `coachLevel`/`canCoachMinLevel`/`canCoachMaxLevel` each in `[0, 1000]`
  (spec's scale, matching `estimatedLevel`); `canCoachMinLevel <= canCoachMaxLevel` when both are
  set. Applied identically on create and PATCH (merged-value validation, same pattern as
  `FieldDefinitionValidator`).
- **`maxGroupsPerDay`/`maxGroupsPerWeek`**: validated `>= 1` when present, and `maxGroupsPerDay <=
  maxGroupsPerWeek` when both are present — a reasonable sanity constraint invented for this
  milestone (the spec doesn't specify one); documented here since it's not spec-mandated.
- **Availability is full-replace ("set") semantics**: `PUT .../availability` with a list of
  `{timeSlotId, kind}` deletes all existing `coach_time_slot` rows for that coach and inserts exactly
  what was sent — a time slot omitted from the request is "unlisted". **Unlisted/neutral is treated
  as AVAILABLE** everywhere that consumes availability (M5 review fix 2), matching the solver design
  where CoachFact carries only `unavailableTimeSlotIds` (docs/design/04-solver.md): only an explicit
  `UNAVAILABLE` row blocks a coach from a slot; `AVAILABLE` is an explicit confirmation; `PREFERRED`
  additionally feeds M6b's coach-preference soft constraint. `kind` must be one of
  `AVAILABLE`/`UNAVAILABLE`/`PREFERRED`; every `timeSlotId` must belong to the same plan; duplicate
  `timeSlotId`s within one request are rejected with a clear 400 rather than surfacing as a raw
  `UNIQUE` constraint 409.

## Capacity analysis (spec §12.4)

`GET /api/plans/{planId}/capacity` → `CapacityService.compute`. Reproduces the spec's own worked
example **exactly** (asserted in `CapacityServiceTest.reproducesSpecSection12Point4ExampleExactly`
and independently re-verified against the running jar — transcript below): 122 anmälda, 11 active
TrainingBlocks, target 10 / max 12 → targetkapacitet 110, maxkapacitet 132, status *"Möjligt, men
grupperna blir större än target"*.

- **`participantCount` = "Antal anmälda"** (M5 review fix 3): ALL registered participants,
  including already-waitlisted ones (a waitlisted person is still anmäld); the waitlisted subset is
  reported separately as `waitlistedCount` so the UI can show both. The §12.4 example numbers are
  unaffected (no waitlisted participants there).
- `waitlistRisk` has a 4th value beyond the spec's implied 3 (`NONE`/`OVER_TARGET`/`OVER_MAX`):
  **`UNKNOWN`**, when the plan has no configured `defaultGroupTargetSize`/`defaultGroupMaxSize` yet —
  returns `null` capacity numbers with an explanatory message instead of crashing or fabricating a
  target. The `NONE` and `UNKNOWN` message texts are this milestone's own invention (the spec only
  gives the `OVER_TARGET` and (implicitly) `OVER_MAX` wording); the two spec-quoted messages are
  reproduced verbatim, including the em dash in *"Fler anmälda än maxkapacitet — kölista krävs"*.
- **`coachShortageRisk` is an MVP estimate, not exact** (documented in `CapacityService`'s Javadoc):
  a true feasibility check is a bipartite-matching problem (coach × slot availability, subject to
  per-coach daily/weekly caps) which is out of scope for a pre-solve *estimate* endpoint. Two
  independent, complementary signals are combined instead:
  1. **Per-slot signal** — for any time slot with active blocks, are there at least as many coaches
     NOT marked `UNAVAILABLE` at that slot as there are active blocks (M5 review fix 2:
     neutral/unlisted counts as available, aligned with the solver's CoachFact semantics)? Catches
     "coaches exist, but explicitly can't make the times that need them" — without false-positive
     warnings for coaches who simply haven't filled in availability yet (regression-tested:
     `noShortageWarningWhenCoachesHaveNoAvailabilityRowsAtAll`).
  2. **Aggregate signal** — sum each coach's effective weekly capacity (`maxGroupsPerWeek` if set,
     else `maxGroupsPerDay × distinct days with active blocks` if set, else "unlimited", modeled as
     the total active-block count — a safe sentinel) against `groupsRequiringCoachEstimate`. Catches
     "coaches are at the right times, but their combined max-groups limits are too low" (brief's
     "maxGroups limits" scenario).
  Both are exercised as independent scenarios in `CapacityServiceTest` (each isolated so only one
  signal fires) and combined in the E2E transcript below (both fire together there).
- `groupsRequiringCoachEstimate = activeTrainingBlockCount` (MVP: 1 coach per group/block, per the
  brief).
- `perTimeSlot` lists every time slot in the plan (not just ones with active blocks) with its active
  block count, not-unavailable coach count (`coachesAvailableCount`), and explicitly-preferred coach
  count (`coachesPreferredCount` — kept separate as a forward signal for M6b's coach-preference soft
  constraint) — the data the per-slot shortage signal (and a future Kapacitetsvyn table) needs.

## Test-isolation note (for future milestones touching club-level data)

`venue`/`court` are club-level (no plan scoping), and every `@SpringBootTest` test *class* shares one
Spring context/DB across all its `@Test` methods (the `@TempDir` that keys the context is `static`).
A test asserting "no venue exists yet" or "exactly N venues exist" is therefore only valid if it's
the *first* thing to touch that table in its class — which the M1-era convention of "every test
scopes assertions to the specific ids it created" avoids by construction, but the M5 "auto-create
default venue" behavior can only be asserted via a global (not id-scoped) check. Fix applied:
`DefaultVenueAutoCreationTest` is a separate, `@TestMethodOrder`-pinned test class with its own
`@TempDir` (⇒ its own context/DB) specifically for that scenario, rather than a method inside the
general `VenueCourtControllerTest`. `CapacityServiceTest`'s per-test court names also needed a
monotonic counter rather than a UUIDv7 substring: UUIDv7's first 8 hex characters are a millisecond
timestamp prefix that's constant for ~65 seconds (RFC 9562 §5.7), so `Uuid7.generate().substring(0,
8)` is *not* unique within one fast test run — a real bug caught by the new
`court(venue_id, name)` unique index during development.

## Known MVP limitations (M5 review fix 7 — deliberate, with rationale)

- **`court.active` is display-only**: whether a (slot, court) pair is schedulable is decided solely
  by `training_block.active` — the block is the scheduling truth the solver (M6) will consume.
  `court.active` exists per spec §7.8 and is round-tripped by court CRUD, but nothing derives block
  state from it in MVP: deactivating a court club-wide would need a fan-out rule over every plan's
  blocks (and an answer to "what happens when it's reactivated?") that spec §12.3 explicitly scopes
  down to per-block deactivation instead.
- **Court-rename pool caveat**: the generator identifies pool courts by the exact name pattern
  `^Bana (\d+)$`. Renaming an auto-created court (e.g. "Bana 2" → "Centercourt") via
  `PATCH /api/courts/{id}` removes it from the pool: its existing blocks keep working and stay
  untouched by regeneration (by design — see above), but the next `PUT ...:courts {count:N}` will
  create a fresh "Bana 2" court alongside it. Acceptable for MVP because the council's real-world
  resources genuinely are "Bana 1..4"; a rename feature would need a stable pool-index column
  instead of name parsing.
- **Overnight slots unsupported**: `startTime < endTime` is enforced within one calendar day, so a
  slot like 23:00–01:00 is rejected (400). Beach volleyball training does not cross midnight; M6's
  overlap math (TimeKey) also assumes same-day intervals, so allowing it here would just move the
  breakage downstream.
- **Label-freeze edge**: "was the label customized?" is inferred by comparing the stored label to
  what auto-generation would produce for the slot's *current* day/times (no separate tracking
  column). Consequence: if a user hand-types a label that exactly matches the auto-generated text,
  it is indistinguishable from an auto label and will keep regenerating on later day/time changes.
  Harmless (the "custom" text was identical anyway) and much simpler than schema-tracking
  customization state; explicit `label` in a PATCH always wins, explicit `null` always regenerates.
- **Coach-shortage estimate is heuristic, not exact** (see "Capacity analysis"): the two combined
  signals can miss cross-slot assignment conflicts that only a real matching would catch (e.g. one
  coach needed at two overlapping slots simultaneously); the authoritative feasibility check is the
  M6 solver itself. The estimate is deliberately conservative in the other direction (no
  false-positive warnings from missing availability data).
