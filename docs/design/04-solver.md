# Design: Timefold Solver Module (verified)

> Generated by design agents 2026-07-01, web-verified. NOTE: apply the fixes from 05-solver-verification.md (3 major, 8 minor) — they are the contract amendments.

## Executive summary

Complete solver-module design for the beach volleyball group-assignment tool on Timefold Solver Community 1.33.0 (pinned). Three planning entities (PlayerAssignment with allowsUnassigned group, GroupSchedule with trainingBlock, CoachSlot with allowsUnassigned coach) give per-aspect @PlanningPin locking for free and make waitlist + coach shortage first-class, never-hard-violating states. HardMediumSoftLongScore with medium reserved for unassignment (scaled by Prioritet); all level math in fixed-point x100 integers with ArchUnit-enforced no-float rule. All 24 spec constraints designed with Constraint Streams sketches, justifyWith payloads and ConstraintVerifier tests (10.2/10.23 satisfied by construction). Runtime weights/level-reclassification via ConstraintWeightOverrides (zero = disabled, verified). Determinism: NO_ASSERT + randomSeed 0 + sorted input for user solves; step-count termination for CI golden scores on macOS+Windows. Explainability via SolutionManager.analyze(FETCH_ALL) + ScoreAnalysis.diff what-ifs, lazy-cached per-player records whose comparison set always includes friend-wish/coach-wish/previous groups, plus on-demand 'Varför inte grupp X?'. Enterprise boundary confirmed clean; Timefold's free nonprofit license recorded as 2.x upgrade path. Implementation split into 3 agent-sized milestones with test gates.

## Key decisions

- **Split the spec's two-variable Group entity into three planning entities: PlayerAssignment (var: group), GroupSchedule (var: trainingBlock), CoachSlot (var: coach); Group itself becomes an immutable problem fact**
  - Rationale: @PlanningPin is entity-level in 1.x (verified: 'applies to all the planning variables of that planning entity'), and spec §15.2/§15.3 require locking tid/bana and tränare independently. Splitting gives exact per-aspect pinning with zero custom move-filter code, makes requiredCoachCount=0 trivial (no CoachSlot created) and requiredCoachCount>1 natural (one slot per required coach), at the acceptable cost of joins against GroupSchedule in time-dependent constraints.
- **Waitlist modeled as @PlanningVariable(allowsUnassigned = true) on PlayerAssignment.group with a MEDIUM-level unassignment penalty scaled by the Prioritet field; HardMediumSoftLongScore adopted from day one**
  - Rationale: Overcapacity is the normal case. Hard must stay pure feasibility so 'plan har 8 på kölistan' is never confused with 'plan bryter regler'; medium makes minimizing unplaced players lexicographically dominate all soft quality without competing with hard. A player who can attend no slot is left unassigned (medium cost) instead of violating TimeAvailabilityHard — exactly the required behavior. Spec §9.4/§16.4 explicitly wants a medium-capable model.
- **Runtime constraint weights and hard/medium/soft reclassification via ConstraintWeightOverrides field on the solution (verified in 1.33), not @ConstraintConfiguration**
  - Rationale: ConstraintWeightOverrides.of(Map<String,Score_>) replaces a constraint's whole weight Score by constraint name — including its LEVEL — so §9.4's per-field hard/soft choice is just ofHard(w) vs ofSoft(w); zero weight is documented to remove the constraint from processing (verified javadoc), giving enable/disable for free. @ConstraintConfiguration is the legacy mechanism and does not support level changes.
- **All constraint math in fixed-point integers (levelScaled = round(level*100)); LevelBalance = sum of absolute deviations from floorDiv-mean, GroupOrderByLevel compares means by cross-multiplication; ArchUnit test bans float/double/BigDecimal in solver.domain and solver.constraints packages**
  - Rationale: Cross-platform score equality is a build-breaking CI gate; IEEE-754 double accumulation order and BigDecimal-free integer arithmetic are the only safely bit-identical options. The built-in loadBalance collector was verified to return BigDecimal unfairness (six decimals) — usable but hand-rolled SAD is more explainable ('spridning 82 → 141') and keeps the no-float rule absolute. Overflow headroom verified: worst-case soft total ~1e10 vs long max 9.2e18.
- **Determinism split: user solves run NO_ASSERT + randomSeed 0 + wall-clock presets (10/60/120 s) injected per job via SolverJobBuilder.withConfigOverride; CI golden-score tests run PHASE_ASSERT with stepCountLimit/unimprovedStepCountLimit termination and assert exact score equality across macOS/Windows runners**
  - Rationale: REPRODUCIBLE and FAST_ASSERT are deprecated since 1.20 (verified); NO_ASSERT is the fast reproducible mode, PHASE_ASSERT the reproducible default with checks. Time-based termination cannot reproduce across hardware, so golden tests must terminate on step counts (verified TerminationConfig fields). Default local search is Late Acceptance (verified in DefaultLocalSearchPhaseFactory), which unlike Simulated Annealing has no time-gradient dependency.
- **Explainability: one SolutionManager.analyze(solution, FETCH_ALL) per run persisted at plan level; per-player ExplanationRecords computed lazily on request and cached by (runId, planRevision); comparison set = union of friend-wish groups, coach-wish groups, previous-term group, top-3 by what-if score delta; separate on-demand 'why not group X' endpoint; what-if = rebuild-solution-from-state + apply move + analyze + ScoreAnalysis.diff (verified)**
  - Rationale: Each what-if costs one FETCH_ALL analyze of a ~150-entity solution (tens of ms); eager precomputation for 130 players x ~5 candidates ≈ 650 analyses per run is wasted work, while lazy per-click is ~5 analyses (<0.5 s). The union rule guarantees 'varför inte gruppen med kompisen?' is always answerable, which top-N alone cannot. Rebuilding the solution via the assembler avoids depending on non-public SolutionCloner internals.
- **Group generation, estimatedLevel derivation, greedy baseline and season-wide conflict detection are plain deterministic Java services outside the solver**
  - Rationale: Spec §11 explicitly wants level logic computed, stored and shown (not solved); §19.2/§14.4 conflict warnings must cover ALL plans in a season regardless of status, which is a reporting query, not an optimization constraint; the greedy baseline (§16.7) doubles as fallback and as the golden-test sanity reference. Only in-solve blocking (§10.24) enters the solver, as pre-assembled SavedPlanResourceUsage facts.

## Risks

- **The versioned docs URL https://docs.timefold.ai/timefold-solver/1.33.x/... returns 404 — only /latest/ (2.x) is hosted, so the implementing agent could accidentally read 2.x docs and use removed/renamed APIs** → All API names in this design were verified against the v1.33.0 GitHub source tag (raw.githubusercontent.com/TimefoldAI/timefold-solver/v1.33.0/...). The design document instructs the coding agent to treat the source tag and local javadoc JARs as the reference, never /latest/; every class in the design carries its exact package path.
- **Adjacent-groups-only GroupOrderByLevel and mean-comparison via cross-multiplication can leave non-adjacent inversions unpenalized transitively when a middle group is empty or when means are equal at integer resolution** → Empty groups are excluded by the count>0 filter and separately penalized by GroupMinSize/GroupSizeTarget; SAD-based LevelBalance plus PreviousGroupContinuity pull in the same direction, and the golden-dataset tests include an ordering assertion. If real data shows inversions, widen to order-distance-weighted pairs (o, o+2) — a localized constraint change.
- **Deterministic tie-breaking among equal-priority waitlisted players is 'reproducible but arbitrary' (fixed seed + sorted input), which may be perceived as unfair** → The Prioritet field gives kansliet explicit control; the unassignment justification carries priority and the per-player explanation shows what-if deltas for every group, so the arbitration is visible. If needed later, add a small deterministic tiebreak term (e.g. earlier registration order) as a documented soft constraint — the medium/soft split leaves room.
- **Splitting Group into GroupSchedule + CoachSlot forces joins on GroupSchedule in every time-dependent constraint, adding Constraint Streams node count and some incremental-recalculation cost** → Problem size is tiny (≤150 entities, ≤16 blocks, ≤20 coaches); Late Acceptance at this scale runs thousands of steps/second even with the extra joins. The M-S1 gate includes a solve-throughput smoke check on the 120-player dataset; if it ever mattered, groupBy caches per-group tuples so the join fan-out stays O(groups).
- **ConstraintVerifier tests use default weights, while production behavior depends on user-supplied ConstraintWeightOverrides; a mis-mapped override (wrong constraint name string) would silently zero a constraint** → Constraint keys are declared once in ConstraintMetadata and used both by .asConstraint(...) and by the override assembler; a dedicated test asserts ConstraintWeightOverrides.getKnownConstraintNames()-driven map keys all resolve against the provider's constraint set, and the solve endpoint rejects unknown keys at assembly time.
- **What-if and explanation analyses run concurrently with a possible active solve on the same plan and could read half-updated state** → SolveCoordinator serializes per-plan: what-if/explanations operate on the last persisted run snapshot (assembler rebuilds from DB, never from live solver memory), and endpoints return 409 SOLVING_ACTIVE for mutating operations while getSolverStatus(problemId) reports SOLVING_ACTIVE.

## 1. Planning model

### 1.1 Decision: three planning entities, Group is a problem fact

The spec (§16.1) suggests `PlayerAssignment` (var: group) and `Group` (vars: trainingBlock + coach). We instead split Group's two variables into **two single-variable planning entities**, because:

1. **Locks (spec §15.2/§15.3) must be independent per aspect.** `@PlanningPin` is entity-level in 1.x (verified: *"applies to all the planning variables of that planning entity"*). With a two-variable Group entity, pinning the time would also pin the coach. Splitting gives exact per-aspect pinning with zero custom move filters. (Full analysis in §5.)
2. **`requiredCoachCount = 0`** — no CoachSlot entity is created, so no null-coach special case pollutes the coach constraints. `requiredCoachCount > 1` — one CoachSlot per required coach.
3. Group's static attributes (name, order, sizes, level band) become an immutable problem fact, which constraint code and explanations reference safely.

Cost: constraints that need "the group's time" must join `GroupSchedule` on group. Acceptable at this scale (≤150 entities).

### 1.2 Entity and fact classes (`se.klubb.groupplanner.solver.domain`)

All solver-side classes are **plain POJOs assembled from the DB by `SolverInputAssembler`** — never persistence entities. All IDs are `long` (stable DB ids). All numeric planning data is `int`/`long` (see §3).

```java

    @PlanningId
    private long id;                        // = participantProfileId
    // --- problem facts folded in (immutable during solve) ---

    private String displayName;

    private int priority;                   // Prioritet standard field, 1..5, default 3
    private Integer previousGroupOrder;     // nullable; mapped from previousGroupName (§8)
    private long[] unavailableTimeSlotIds;  // sorted; from 'Kan inte tider'
    private long[] preferredTimeSlotIds;    // sorted; from 'Föredrar tider'
    // --- planning ---
    @PlanningVariable(allowsUnassigned = true)   // verified 1.33 syntax; 'nullable' is deprecated
    private Group group;                    // null == Oplacerad/Kölista
    @PlanningPin
    private boolean pinned;                 // spec §15.1 player lock
    // helpers used by constraints (pure int/boolean, no floats):
    public boolean canAttend(long timeSlotId) { /* binary search unavailable ids */ }
    public boolean prefers(long timeSlotId) { ... }
}

@PlanningEntity
public class GroupSchedule {
    @PlanningId
    private long id;                        // = groupId
    private Group group;                    // fact
    @PlanningVariable                       // NOT unassignable: see 1.4
    private TrainingBlock trainingBlock;
    @PlanningPin
    private boolean pinned;                 // spec §15.2 tid/bana lock
}

@PlanningEntity
public class CoachSlot {
    @PlanningId
    private long id;                        // synthetic: groupId * 10 + slotIndex
    private Group group;                    // fact
    private int slotIndex;                  // 0..requiredCoachCount-1
    @PlanningVariable(allowsUnassigned = true)   // coach shortage must not explode into double-bookings
    private CoachFact coach;
    @PlanningPin
    private boolean pinned;                 // spec §15.3 tränare lock
```

### 1.3 Planning solution


Since each variable's range is homogeneous, the `valueRangeProviderRefs` can be left implicit (single provider per type); keep the explicit ids anyway for clarity: `@PlanningVariable(allowsUnassigned = true, valueRangeProviderRefs = "groupRange")` etc.

**Shadow variables: none.** Group size, level sums etc. are computed inside constraints via `groupBy` and post-solve in plain Java for display. This keeps the model minimal and the clone problem trivial.

### 1.4 Why `trainingBlock` is NOT unassignable while `group` and `coach` are

- `group` (player): overcapacity is the *normal* case → waitlist is a first-class value (`null`), penalized at medium (§2).
- `coach`: coach shortage is common; if the variable required a value, the Construction Heuristic would double-book coaches and produce misleading hard violations. With `allowsUnassigned = true`, a missing coach shows up as exactly one clean `CoachRequirementHard` violation ("Grupp 7 saknar tränare"), which capacity analysis predicts.
- `trainingBlock`: group generation (§7) guarantees `groupCount ≤ activeTrainingBlockCount`, so a perfect matching always exists; `GroupRequiresTrainingBlock` (10.2) is thereby **satisfied by construction** and needs no constraint code. If cross-plan blocking makes some blocks undesirable, that is expressed as hard constraints (10.24), not by shrinking the range — the "visa konflikter men tillåt ändå" mode (§14.4) needs the blocks selectable.

## 2. Waitlist mechanics (Oplacerad/Kölista)

### 2.1 Score design

Constraint `unassignedPlayer` (MEDIUM):

```java
Constraint unassignedPlayer(ConstraintFactory f) {
    return f.forEachIncludingUnassigned(PlayerAssignment.class)   // verified: forEach() EXCLUDES null-var entities
            .filter(pa -> pa.getGroup() == null)
            .penalize(HardMediumSoftLongScore.ofMedium(100),
                      PlayerAssignment::getPriority)               // matchWeight = priority 1..5
            .justifyWith((pa, score) -> new UnassignedPlayerJustification(
                    pa.getId(), pa.getDisplayName(), pa.getPriority()))
            .indictWith(pa -> List.of(pa))
            .asConstraint("unassignedPlayer");
}
```

- Score level **medium**: lexicographically below every hard constraint, above every soft one. Consequences:
  - A group at `maxSize` never gets a 12th player to save a friend wish (hard > medium > soft), and the solver **prefers waitlisting over breaking `GroupMaxSizeHard` or `TimeAvailabilityHard`** — a player who can attend no offered slot ends up `null` with only a medium cost, never a hard violation.
  - Soft quality (level balance, wishes) can never justify leaving someone unplaced.
- **Prioritet scaling**: matchWeight = `priority` (1..5, default 3, from the standard field §9.2). Unassigning a priority-5 player costs `500 medium`, a priority-1 player `100 medium` → under capacity pressure the solver deterministically sheds lowest priority first. Ties are resolved reproducibly (fixed seed + sorted input) but arbitrarily; this is documented and visible in the explanation.
- The **plan-level summary** reports `unassignedCount` and the medium score separately from feasibility (`hardScore == 0` still means "no rules broken", per `HardMediumSoftLongScore.isFeasible` semantics).

### 2.2 Interaction contract (tested)

| Situation | Outcome | Test |
|---|---|---|
| 131 players, capacity 130 (12×max) | exactly the lowest-priority player unassigned; hard = 0 | `WaitlistOverCapacityTest` |
| Player's `unavailableTimeSlotIds` covers all block times | player unassigned; `timeAvailabilityHard` has 0 matches | `WaitlistNoFeasibleTimeTest` |
| Player pinned to a group over max | hard violation stays visible (pin wins over solver, user owns it) | `PinnedOverflowTest` |
| Pinned with `group == null` | allowed: an explicit "lås till kölista" | `PinnedWaitlistTest` |

### 2.3 "Varför hamnade jag på kölistan?"

Explanation path (§11): for an unassigned player the ExplanationService runs the standard comparison set (§11.3) against **every** group (12 what-if probes, still < 1 s) and reports per group the first blocking reason in severity order: `GroupMaxSizeHard` (full: 11/11), `TimeAvailabilityHard` (kan inte tiden), `DifferentGroupHard`, else "plats fanns men score försämras med X (lägre prioritet än de placerade)". The `UnassignedPlayerJustification` provides the headline; the probes provide the per-group rows. This makes the waitlist fully explainable with the same machinery as normal placements.

## 3. Score type and integer discipline

### 3.1 Score type: `HardMediumSoftLongScore`

`ai.timefold.solver.core.api.score.buildin.hardmediumsoftlong.HardMediumSoftLongScore` (verified: `of(long,long,long)`, `ofHard/ofMedium/ofSoft`, `hardScore()/mediumScore()/softScore()`).

- **Hard** = feasibility only (rule violations kansliet must see as red).
- **Medium** = unassignment (waitlist) only, by convention. Reserving the level keeps "8 på kölistan" categorically separate from quality trade-offs.
- **Soft** = all weighted quality.
- `Long` variant: headroom analysis in §3.4. Spec §9.4/§16.4 asks for a medium-capable model — this delivers it while MVP UI may still present only Hard/Soft plus the waitlist count.

### 3.2 Fixed-point scaling

- DB stores `estimatedLevel`, `manualLevelScore`, `coachLevel`, `canCoachMin/Max` as REAL on the **0–1000 level scale** (§8).
- The assembler converts once: `levelScaled = (int) Math.round(estimatedLevel * 100.0)` → range 0..100 000. `Math.round(double)` is IEEE-754-defined and bit-identical cross-platform on Java 21; this is the **only** place floating point touches level data.
- **Rule: no `float`, `double`, or `BigDecimal` anywhere in `se.klubb.groupplanner.solver.domain` and `se.klubb.groupplanner.solver.constraints`.** Display values (e.g. "nivåsnitt 637,4") are formatted from scaled ints in the explain layer.
- Enforcement test (ArchUnit, part of M-S1 gate):


### 3.3 LevelBalance spread — exact integer formula

For group *g* with members' scaled levels `x_1..x_n` (n ≥ 1):

```
sum_g    = Σ x_i                       (long)
mean_g   = Math.floorDiv(sum_g, n)     (floor division – defined, deterministic)
SAD_g    = Σ |x_i − mean_g|            (sum of absolute deviations, scaled units)
penalty  = Math.floorDiv(SAD_g, 100)   (back to whole level points)
```

`LevelMath.sadPoints(int[] levelsScaled)` is a pure static function; the same function feeds constraint matchWeight, group-level display ("nivåspridning 82") and what-if deltas ("82 → 141"), so numbers agree everywhere. The built-in `ConstraintCollectors.loadBalance` exists in 1.33 Community but its `LoadBalance.unfairness()` returns `BigDecimal` (verified) — rejected to keep the integer rule absolute and the metric explainable.

### 3.4 Overflow headroom (130 players)

Worst cases per level, with max UI weight 10 000:

- Soft: LevelBalance ≤ 12 groups × 11 000 points × 10 000 = **1.3e9**; pair wishes ≤ ~1 300 matches × 10 000 = 1.3e7; ordering ≤ 11 pairs × 1 000 × 10 000 = 1.1e8. Total ≪ 1e10.
- Medium ≤ 130 × 5 × 100 = 6.5e4. Hard ≤ ~10 000 × 10 000 = 1e8.
- `Long.MAX_VALUE ≈ 9.2e18` → headroom factor > 1e8. A guard test (`ScoreHeadroomTest`) computes the analytic worst case from metadata and asserts < 1e15.

## 5. Locks (spec §15)

**Chosen mechanism: entity splitting + `@PlanningPin` (option A).** Alternatives considered:

- (B) *Per-variable lock facts + hard constraints*: keeps one Group entity but locks become score-level fictions — the solver wastes moves proposing them, explanations get polluted with "LockViolation" matches, and §23.6 "kan inte ändras av ny solve" is only statistically true. Rejected.
- (C) *Custom move filters*: 1.x local search + CH would need coordinated filtering across move types; error-prone, unverifiable cheaply. Rejected.
- (A) gives the guarantee mechanically: `@PlanningPin` (verified entity-level, all variables of the entity) on `PlayerAssignment`, `GroupSchedule`, `CoachSlot` — and because tid/bana and tränare live on *different* entities, spec §15.2 and §15.3 are independently lockable. 10.23 becomes satisfied-by-construction.

Implementation details:
- DB rows `PlayerAssignment.locked`, `Group.assignedTrainingBlockLocked` (new column or reuse `TrainingBlock.locked` semantics — recommend explicit `group_schedule_locked` on Group), `CoachAssignment.locked` map 1:1 to the pin booleans; the assembler sets the current value and `pinned=true`.
- Locking a coach to a group with several slots pins the **lowest-index free slot** (deterministic).
- §15.5 "Optimera endast X" is realized by pinning whole entity classes: e.g. "optimera endast spelare" → assembler pins all GroupSchedules + CoachSlots at their current values.
- Pre-solve validator: every pinned entity must have a value present in its value range (or null for pinnable-to-waitlist players); violations are rejected with a 422 before solve, so the solver never sees an impossible pin.
- Tests: `LockRespectTest` (all three lock kinds survive a full solve unchanged), plus a pin+re-solve golden test in CI.

## 6. Cross-plan blocking and season conflicts

### 6.1 TimeKey matching rule (shared, single implementation)

`TimeKey.overlaps(other)` in `se.klubb.groupplanner.common.time`:

1. If **both** have a concrete date (`epochDay >= 0`): overlap ⇔ `epochDay == other.epochDay && startMinuteOfDay < other.endMinuteOfDay && other.startMinuteOfDay < endMinuteOfDay`.
2. Otherwise (at least one date-less, per spec §7.9 TimeSlot may carry only dayOfWeek): overlap ⇔ `dayOfWeek == other.dayOfWeek && intervals intersect` (same formula).

Pure int arithmetic; unit-tested exhaustively (boundary touching 19.30/19.30 = no overlap).

### 6.2 SavedPlanResourceBlock feed (in-solver)

`SolverInputAssembler.collectSavedPlanUsages(activityPlanId, BlockingOptions opts)`:

- Source plans: all ActivityPlans in the same SeasonPlan with status `saved | locked | published`, excluding the plan being solved.
- Per source plan, for each persisted assignment triple (group → block, player → group, coach → group), emit:
  - `PERSON(personId, timeKey)` for every **player** (if `opts.blockPlayers`) and every **coach** (if `opts.blockCoaches`) — coaches emitting PERSON usages is what makes 10.17 hold across plans.
  - `COURT(courtId, timeKey)` for every used block (if `opts.blockCourts`).
- `opts` = the §14.4 checkboxes, stored per solve request.
- **"Visa konflikter men tillåt ändå"**: usages are still emitted, but the solve request maps the three `savedPlan*` constraints to `ofSoft(1)` via ConstraintWeightOverrides (level reclassification, §10). Matches then still exist → shown as warnings, never zeroed out (zero would erase the explanations, verified `ConstraintWeightOverrides` zero semantics).

### 6.3 ConflictService (outside the solver)

`se.klubb.groupplanner.season.ConflictService.findConflicts(seasonPlanId)`:

- Loads persisted assignments of **all** plans in the season **regardless of status** (also drafts — §19.2 Säsongsvyn warns early; §14.4 mode needs it).
- Builds `ResourceUsage(planId, planName, kind PLAYER|COACH, personId?, courtId?, groupName, TimeKey)` rows, sorts by stable ids, sweeps for pairwise overlaps per personId and per courtId using the same `TimeKey.overlaps`.
- Output `List<SeasonConflict>`: `{type: PERSON_DOUBLE_BOOKED | COACH_PLAYS_WHILE_COACHING | COURT_DOUBLE_BOOKED, personId?, courtId?, usages: [ {planId, planName, groupName, timeLabel} ], severity: WARNING}` — pure reporting, feeds Säsongsvyn and the post-solve conflict panel. No Timefold dependency.

## 7. Group generation (pre-solve)

`GroupGenerator.generate(activityPlanId)` — plain deterministic service, runs when the user creates/refreshes groups:

```
P  = active participant count
T  = defaultGroupTargetSize, X = defaultGroupMaxSize, B = active TrainingBlock count
groupCount = clamp(ceil(P / T), 1, B)
expectedWaitlist = max(0, P − groupCount * X)   // surfaced in capacity analysis (§12.4)
```

- **Naming**: `<prefix> <n>` where prefix = ActivityPlan.category if set else plan name ("Herr 1" … "Herr 12"); `groupOrder = n`, 1 = highest level.
- **Sizes**: min/target/max from ActivityPlan defaults; user-editable per group afterwards.
- **Initial level ranges** (informational, never hard): sort participants by `(levelScaled DESC, participantProfileId ASC)`, slice into `groupCount` contiguous slices as even as possible (sizes differ by ≤1); `levelMinScaled/levelMaxScaled` = slice min/max, group 1 gets the top slice. These bands seed `coachLevelFit` display and the UI, and give the greedy baseline its assignment.
- **Re-generation when groups exist**: if any PlayerAssignment/CoachAssignment lock or a `saved+` plan references the groups → refuse with explicit "Återställ grupper" confirmation (destructive, clears assignments). Otherwise: keep group ids where `groupOrder` matches (preserves explanations/history), add/remove tail groups to reach the new count, recompute names/bands. Deterministic; unit-tested with shrink/grow cases.

## 8. Level logic (spec §11) — plain service, not solver logic

`LevelService.recomputeEstimatedLevels(activityPlanId)` runs at import time and on demand; results are **stored** on ParticipantProfile (REAL `estimatedLevel` on the 0–1000 scale, `levelConfidence` REAL 0–1, `levelSource` TEXT) and **shown** in the UI. The solver only ever sees the precomputed `levelScaled` int.

Derivation chain (spec §11.2), first hit wins:

| Priority | Source | estimatedLevel | confidence | levelSource |
|---|---|---|---|---|
| 1 | `manualLevelScore` present | the value (clamped 0–1000) | 1.0 | MANUAL |
| 2 | `rankingPoints` present | **identity normalization**: level scale is defined as the 0–1000 ranking scale, so `estimatedLevel = clamp(rankingPoints, 0, 1000)` (spec's own example uses "nivåscore 640" against ranking data; no distortion, fully explainable) | 0.8 | RANKING |
| 3 | `previousGroupLevel`/`previousGroupName` present | previous-group mapping below | 0.5 | PREVIOUS_GROUP |
| 4 | none | `estimatedLevel = 500` (midpoint), `manualReviewFlag = true` | 0.2 | NONE — "flagga för manuell bedömning" |

**Previous-group → level mapping**: parse trailing integer from `previousGroupName` (regex `(\d+)\s*$`, e.g. "Herr 7" → 7; unparsable → chain step 4 + flag). With `G_prev` = max parsed index seen in the import (per plan), map group *g* to the midpoint of its band on the 0–1000 scale, higher group = higher level:

```
estimatedLevel(g) = round(1000 * (G_prev − g + 0.5) / G_prev)     // g=1, G=12 → 958; g=12 → 42
previousGroupOrder(pa) = g                                        // consumed by 10.8 directly
```

`levelConfidence < 0.6` players are listed in the result view ("Osäkra spelare"), per §11.4.

### 9.3 Termination

- **CI / golden-score tests** — deterministic termination only (verified `TerminationConfig` fields `stepCountLimit`, `unimprovedStepCountLimit`, `moveCountLimit`): build via `SolverConfig.createFromXmlResource("solverConfig.xml").withEnvironmentMode(EnvironmentMode.PHASE_ASSERT).withTerminationConfig(new TerminationConfig().withStepCountLimit(2000))` and `SolverFactory.create(config)`. Time-based termination is forbidden in tests because step counts, not seconds, are hardware-independent.
- **Determinism inputs**: fixed `randomSeed 0`; `SolverInputAssembler` sorts every list by stable id before building the solution (participants by profileId, groups by groupOrder, blocks by (timeSlotId, courtId), coaches by coachProfileId, wish facts by (fieldDefinitionId, aId, bId)); stable IDs come from the DB, never from row order or iteration order (spec §4.5).
- **CI gate**: committed anonymized datasets (10/20/120 players, spec §22.2) run on `macos-latest` and `windows-latest`; the job asserts **exact equality** of `score.toString()` and of the hard-violation count against committed golden values, and equality between the two OS runs. Any drift fails the build.

### 9.4 Async solving (SolverManager)

`SolveCoordinator` (`se.klubb.groupplanner.solver.run`):

```java
SolverJob<GroupPlanSolution, Long> job = solverManager.solveBuilder()
    .withProblemId(activityPlanId)
    .withProblem(assembler.assemble(activityPlanId, request))
    .withBestSolutionConsumer(best -> progressRegistry.update(activityPlanId, best))  // score ticks
    .withFinalBestSolutionConsumer(best -> runService.persistRun(activityPlanId, best))
    .withExceptionHandler((id, t) -> runService.failRun(id, t))
    .withConfigOverride(new SolverConfigOverride<GroupPlanSolution>()
        .withTerminationConfig(presetFor(request.profile())))
    .run();                                                    // all methods verified on SolverJobBuilder
```

- **Progress**: 1.33 Community has no percent-progress API; we report `{elapsedMs, limitMs, bestScore, feasible, unassignedCount, improvementCount}` from the registry + `solverManager.getSolverStatus(problemId)` (`NOT_SOLVING | SOLVING_SCHEDULED | SOLVING_ACTIVE`).
- **Cancel**: `solverManager.terminateEarly(activityPlanId)`; the final-best consumer still fires and persists the best-so-far (spec: användaren ska kunna avbryta).
- One solve per plan at a time; mutating endpoints return 409 while `SOLVING_ACTIVE`.
- **Persisting a run** writes `OptimizationRun` (input snapshot JSON, weights JSON, score string, durations) + assignments + one `analyze(FETCH_ALL)` plan summary (§11.1).

### 9.5 Greedy baseline (spec §16.7)

`GreedyBaselineService` — no Timefold: sort players `(levelScaled DESC, id ASC)`; slice into generated groups (§7); assign blocks to groups by `(groupOrder ASC)` → blocks sorted `(startMinute ASC, courtId ASC)`; assign coaches greedily by best `coachLevelFit` distance then availability, skipping conflicts. Output uses the same persistence shape, is scored via `SolutionManager.update(solution)` for comparison, and serves as fallback and debugging reference.

## 10. Dynamic constraint weights (verified 1.33 mechanism)

**Use `ConstraintWeightOverrides` — not `@ConstraintConfiguration`/`@ConstraintWeight`.** Verified: `ConstraintWeightOverrides<Score_>` interface with `of(Map<String, Score_>)`, `none()`, `getKnownConstraintNames()`; placed as a field on the `@PlanningSolution`; the solver substitutes the constraint's weight by **constraint name**. `@ConstraintConfiguration` still exists in 1.33 but is the legacy path and cannot change score *level*; overrides replace the entire `HardMediumSoftLongScore` weight, so:

- **Weight change**: `map.put("sameGroupSoft", HardMediumSoftLongScore.ofSoft(80))`.
- **Hard/Medium/Soft reclassification (spec §9.4/§9.5)**: the DB `ConstraintWeightConfig(hardOrSoft, weight, enabled)` row maps to `ofHard(w) | ofMedium(w) | ofSoft(w)` — changing level is just choosing the factory. This is how "visa konflikter men tillåt ändå" (§6.2) and field-level hard/soft choices are realized with zero constraint-code branching.
- **Disable**: `enabled=false` → `HardMediumSoftLongScore.ZERO`; verified javadoc: zero-weight constraints are *excluded from processing* as if absent — meaning **no matches, no explanations**. Consequence (documented for UI): a disabled constraint disappears from explanations; if the user wants "visas men påverkar inte", use soft weight 1 instead. The API layer offers both ("Av" vs "Endast varning").
- **Flow**: solve request → assembler loads `ConstraintWeightConfig` rows for the plan + field-builder-derived configs (§9.1 spec) → validates every key against `ConstraintMetadata.allKeys()` (startup test asserts metadata keys == provider constraint names via a `ConstraintVerifier`-built session; unknown keys → 422) → `solution.setConstraintWeightOverrides(ConstraintWeightOverrides.of(map))` → weights are frozen into the run's `constraintWeightsJson` so explanations always show the weights *that applied* (spec §17 "vilka vikter som gällde").
- Constraints keep sensible **defaults in code** (`penalize(ofSoft(60))`) so the verifier tests and a fresh install work without any config rows.

## 11. Explainability engine (spec §17 + chapter)

### 11.1 Pipeline

After each solve (and after greedy baseline): `ScoreAnalysis<HardMediumSoftLongScore> analysis = solutionManager.analyze(solution, ScoreAnalysisFetchPolicy.FETCH_ALL)` (verified names `FETCH_ALL | FETCH_MATCH_COUNT | FETCH_SHALLOW`). `FETCH_ALL` is required — match-level data feeds everything below and `diff` needs it. Persisted per run:

- Plan-level: per-constraint `{key, weightApplied, scoreTotal, matchCount}` (from `ConstraintAnalysis` — verified components `constraintRef, weight, score, matches, matchCount`).
- Match-level: every `MatchAnalysis.justification()` (our typed records, §4) serialized to JSON, indexed by the entity ids inside the records → per-player and per-group lookup without re-analysis.

### 11.2 Three levels (spec chapter)

- **Person**: selected group; fulfilled hard facts (time OK, size OK — derived by *absence* of matches plus positive statements computed from data); broken wishes (this player's id inside `PairWishBroken/CoachWish/TimePreferenceMissed` justifications); level match (playerLevel vs group mean/band, from `LevelMath`); the **alternatives comparison** (§11.3); applied weights.
- **Group**: members, size vs target/max, mean + SAD spread (same `LevelMath` as the constraint, so numbers match the score), coach, block, warnings = all matches indicting the group, members with broken wishes.
- **Plan**: total score (H/M/S), hard-violation list, waitlist, top constraints by |score| (from `ScoreAnalysis.summarize()`-equivalent data, rendered ourselves in Swedish), most problematic groups (sum of |match scores| per indicted group), players flagged for manual review, capacity/coach/time conflicts (ConflictService §6.3).

### 11.3 Alternative-groups comparison set (critic requirement — normative)

For player *p*, `candidateGroups(p)` = **union** of:

1. every group currently containing any person referenced by *p*'s `personRelation` wishes (vill/måste/vill inte spela med) — so *"varför inte grupp C med kompisen?"* is ALWAYS answerable;
2. every group whose assigned coach matches one of *p*'s `coachRelation` wishes;
3. the group whose `groupOrder` equals *p*'s `previousGroupOrder` (previous-term group);
4. top-N (N = 3) of the remaining groups ranked by what-if score delta (best first).

For each candidate: run the what-if diff (§12) of moving *p* there and produce `{groupId, verdict: WOULD_BREAK_HARD | WORSE | BETTER, scoreDelta, newlyBroken[], newlyFixed[], narrative}`; narrative lines are generated from constraint metadata templates in Swedish ("Grupp C var full: 11/11 (hård maxstorlek)", "Nivåspridning skulle öka 82 → 141 (vikt 2/poäng)"). Unassigned players: candidate set = all groups (§2.3).

### 11.4 Cost model and caching decision

One `analyze(FETCH_ALL)` of this solution (~150 entities, ~30 constraints) costs tens of ms; each what-if = rebuild (§12.1, ~ms) + 2 analyze (before is cached) ≈ **≤ 50 ms**. Eager precomputation = 130 players × ~5 candidates ≈ 650 analyses ≈ 30 s of wasted work per run. **Decision: lazy-computation-with-cache.** `ExplanationRecord` (spec §7.18) rows are written on first request, keyed `(optimizationRunId, participantProfileId)`, with `planRevision` stamped; an optional idle background prefetch fills the cache after solve completes (lowest ids first, cancellable). The base run analysis (11.1) is always persisted eagerly.

### 11.5 On-demand "Varför inte grupp X?"

`POST .../whatif/why-not {participantProfileId, groupId}` runs one what-if diff for **any** group the user picks (not just the candidate set) and returns the same per-candidate shape as §11.3 — this is the guaranteed answer path for arbitrary counterfactuals.

### 11.6 Staleness

Every manual move/lock/weight change bumps `ActivityPlan.planRevision` (INTEGER). Explanation responses include `{runId, basedOnRevision, currentRevision, stale: basedOnRevision != currentRevision}`; UI renders **"Inaktuell — baserad på körning N"** when stale. Cached ExplanationRecords are not deleted (history), just flagged; a new solve creates a new runId.

## 12. What-if (spec §18)

### 12.1 Cloning strategy

We do **not** call Timefold's internal `SolutionCloner` (not a supported public entry point for user code in this direction). Instead `WhatIfService` asks `SolverInputAssembler.assembleFromRun(runId)` to rebuild a fresh `GroupPlanSolution` from the persisted run state (facts from DB + assignment map). Deterministic, ~1 ms at this size, and immune to shared-state aliasing. (Planning cloning remains what the solver does internally; `SolutionManager.analyze` does not move genuine variables and we have no shadow variables.)

### 12.2 Evaluate move

```
base    = assembleFromRun(runId)                    // or current working state + planRevision
baseA   = solutionManager.analyze(base, FETCH_ALL)  // cached per (runId, revision)
moved   = assembleFromRun(runId); moved.find(p).setGroup(targetGroup);
movedA  = solutionManager.analyze(moved, FETCH_ALL);
delta   = movedA.diff(baseA);                       // verified ScoreAnalysis.diff, direction this − other
```

Response content (all integer/scaled, formatted in the explain layer):

- `newGroupSizes`: `{fromGroup: n−1, toGroup: m+1, toGroupMax}`.
- `levelSpreadDeltas`: `LevelMath.sadPoints` before/after for both groups ("82 → 141").
- `scoreDelta`: `{hard, medium, soft}` from `delta.score()`.
- `newlyBroken` / `newlyFixed`: walk `delta.constraintAnalyses()`; matches present in the diff with negative (resp. positive) score are mapped through their justification records + `ConstraintMetadata` to human-readable Swedish (`"Bryter hård maxstorlek i Grupp C (12 > 11)"`, `"Kompisönskemål med Lisa uppfylls"`).
- `wouldBreakHard`: `delta.score().hardScore() < 0`.
- Suggested actions (spec §18.2 MVP list): static options rendered from the verdict — flytta ändå / lås & optimera om / behåll.

### 12.3 Recommendation API (spec §18.2 "föreslå lösningar")

`SolutionManager.recommendAssignment(solution, playerAssignment, PlayerAssignment::getGroup[, FETCH_ALL])` is **present in 1.33 Community core** (verified; `recommendFit` deprecated since 1.15). It evaluates all value-range candidates for an *unassigned* entity and returns ranked `RecommendedAssignment`s with score diffs — a perfect fit for a "Föreslå grupper" panel (clear the player's group on a rebuilt solution, call it, render top-3). **Scope: post-MVP-optional** (M-S3 stretch), since §11.3's what-if probes already cover the required answers; but the pinned 1.33 keeps this API available (2.x moved it to the paid Plus edition — a concrete reason the pin matters).
