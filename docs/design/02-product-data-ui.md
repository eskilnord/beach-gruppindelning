# Design: Persistence, Import/Export, UI, Milestones

> Generated by design agents 2026-07-01, web-verified. NOTE: superseded on these points by adversarial review (see 03): repo root is app/, jlink not jpackage, JdbcClient not JPA, Mantine not AntD, Vite 7, NSIS primary, M0 walking skeleton first.
> CANONICAL API PATHS (M1 decision, supersedes §8 where they differ): `/api/plans/{id}` and `/api/seasons/{seasonId}/plans` (not `/api/activity-plans/...`); auth header is `X-GP-Token`. The implemented backend is the source of truth for M2 typegen. field_definition also carries an explicit `column_name` for storage_kind=COLUMN fields (M1 review decision).

## Executive summary

Designed the full MVP plan for the local group-assignment desktop tool per the Swedish spec. The pivotal verified finding: Timefold Solver 2.x (current 2.2.0) moved the score-analysis/explainability APIs to the paid Plus edition, so the design pins Timefold Community 1.33.x (where SolutionManager.analyze/ScoreAnalysis is Apache-2.0 and maintained through 2026), which in turn pins Spring Boot 3.5.16 and Java 21 (Temurin, bundled via jpackage/jlink). Stack: Tauri 2.11 shell spawning a jpackage'd Spring Boot backend on a random localhost port with token auth, React 19 + Vite + Ant Design frontend in Swedish, SQLite (xerial 3.53.2.0) with Flyway 12.9 migrations and JPA. The design delivers a complete SQLite schema for all §7 entities (JSON columns for snapshots/factors), an 8-step generic import wizard that absorbs the real file's messiness via required-field row skipping, person matching, and reusable mapping templates, a field-builder→ConstraintWeightOverrides flow, resources/coaches/capacity services, a ScoreAnalysis-driven three-level explanation and what-if engine, council-layout xlsx/csv export with comments off by default plus anonymized dataset export, and a full REST surface. The MVP is broken into 9 vertical milestones (backend+DB → frontend shell → import → fields → resources → solver → explainability → saved plans/export → packaging), each independently runnable with acceptance checks mapped to spec §23 and mandatory tests including per-constraint ConstraintVerifier tests and a Mac/Windows score-equality regression in CI.

## Key decisions

- **Pin Timefold Solver Community 1.33.x (1.x line), NOT the current 2.2.0**
  - Rationale: Verified via web: starting with Timefold 2.0 (April 2026), explainability — SolutionManager.analyze()/ScoreAnalysis/recommendation APIs — moved to the paid 'Plus' edition. Explainability is this spec's core requirement (§17, §23.7), and the project is a nonprofit local tool, so the open-source 1.x line (score analysis is Community there, introduced in 1.4; quarterly bugfix releases through 2026) is the only viable choice.
  - Rejected: Timefold 2.2.0 Community (would require hand-rolling constraint-match analysis or paying for Plus); OptaPlanner (unmaintained); hand-written solver (violates spec §1).
- **Spring Boot 3.5.16 + timefold-solver-spring-boot-starter 1.33.0 on Java 21 (Temurin)**
  - Rationale: The Timefold 1.x starter targets Spring Boot 3 (Boot 4 support only arrives with Timefold 2.x per GitHub issue #1941). Boot 3.5.16 is the final, most agent-familiar 3.x patch; the backend is localhost-only so OSS-EOL exposure is minimal. Java 21 LTS is supported by both and has mature jlink/jpackage tooling.
  - Rejected: Spring Boot 4.1 (starter-incompatible with Timefold 1.x, Jackson 3 friction with timefold-solver-jackson); Quarkus (fine but less codegen-agent training data for this app shape).
- **Tauri 2.11.x shell spawning a jpackage app-image backend (bundled jlink JRE), random localhost port + token handshake; Electron as explicit fallback gate at packaging milestone**
  - Rationale: Spec §4.2 prefers Tauri; jpackage app-image gives a native launcher per platform satisfying 'no Java install' (§4.3); the shell/resources pattern is documented. Packaging is isolated to the last milestone so a fallback to Electron (spec-sanctioned) costs nothing earlier.
  - Rejected: Electron-first (heavier, but kept as fallback); GraalVM native-image of the backend (Timefold+POI+Hibernate native config is high-risk for codegen agents); Tauri externalBin single-file sidecar (a JVM app is a directory, so it ships as bundle resources instead).
- **SQLite accessed via Spring Data JPA + hibernate-community-dialects, schema owned exclusively by Flyway 12.9 (ddl-auto=validate), UUIDv7 TEXT PKs, JSON TEXT columns only for snapshots/factor lists/options**
  - Rationale: Most mainstream, best-documented persistence stack for coding agents; Flyway officially supports SQLite; explicit V1..V4 SQL migrations keep the schema reviewable and cross-platform-identical; stable UUIDs satisfy spec §4.5.9 (no row-number identity).
  - Rejected: Hibernate ddl-auto=update (non-reviewable, SQLite ALTER limitations); plain JdbcTemplate (more boilerplate for 20+ entities); storing whole entities as JSON documents (kills queryability for capacity/matching).
- **Generic import wizard with required-field row skipping, server-side ImportSession, and persisted import templates — no hardcoded column names**
  - Rationale: Spec §26.1–2 forbids hardcoding to the sample file. Rows lacking any mapped name field are auto-skipped, which generically absorbs the real file's stacked group-metadata column, blank separators, waiting-list and refund rows without any file-specific logic; templates (header-hash keyed) make the second import one click; POI DataFormatter preserves mixed Excel-time/text cells as display text with no automatic interpretation (spec §2.2).
  - Rejected: Section-aware parser tuned to the sample layout (violates §26); client-side parsing (duplicates logic, worse for 10MB files); automatic parsing of 'Tid'/'RankInfo' free text into constraints (explicitly forbidden — council fills structured fields side-by-side instead).
- **Explanations built from Timefold ScoreAnalysis with justifyWith on every constraint, persisted as explanation_record per run; what-if = clone solution, apply move, ScoreAnalysis.diff**
  - Rationale: Spec §17.4 requires explanations from constraint matches/score analysis, not AI guessing; typed justifications carrying participant/group ids let one builder produce person/group/plan-level views and alternative-group comparisons; persisting records keeps explanations stable even after data edits.
  - Rejected: On-demand-only generation (explanations would drift from the run that produced the plan); LLM-generated explanations (explicitly forbidden); indictment-only API (less structured than justifications for per-participant grouping).
- **9 vertical milestones ending in packaging, with per-constraint ConstraintVerifier tests and a Mac+Windows score-equality CI regression**
  - Rationale: Matches spec §24 build order and the delivery constraint that Sonnet agents implement one reviewed, testable increment per session; every milestone is runnable (curl → web UI → installers); §26.13 demands a test per hard constraint and §22.3 demands cross-platform reproducibility, so both are baked into milestone acceptance rather than left to the end.
  - Rejected: Horizontal layering (backend-complete-first — nothing demonstrable for weeks); packaging early (churns while API is unstable); fewer/bigger milestones (exceeds one agent session and weakens review gates).

## Risks

- **Timefold 1.x maintenance ends after 2026; the project is then on a frozen solver line while explainability in 2.x sits behind the paid Plus edition** → Isolate all Timefold API usage in backend/solver + explain packages behind our own interfaces (SolverService, ExplanationBuilder); pin 1.33.x and take quarterly 1.x bugfixes; document the trade-off in the README. If migration is ever forced, the ExplanationBuilder abstraction localizes the rewrite (either hand-rolled match analysis on 2.x Community or a Plus license).
- **Tauri packaging of a JVM backend (Rust toolchain, per-platform app-image, process lifecycle, Windows signing) is the least codegen-friendly part and could stall the final milestone** → Confined entirely to M9 with an explicit decision gate: if the Tauri shell is not booting the packaged backend within the session budget, switch to Electron (spec-sanctioned alternative) whose main-process spawn of a jpackage app-image is trivial. Backend is shell-agnostic (plain localhost HTTP + token), so nothing earlier changes.
- **Cross-platform determinism (§22.3/§23.10) may break via iteration-order or locale differences despite fixed seed and reproducible mode** → Deterministic sort of all entity/fact lists by UUID before solve; forbid HashMap iteration in constraint code (review checklist); normalize locale (fixed sv-SE formatting only at UI/export edges, ISO-8601/plain numbers internally); CI matrix job compares score and hard-violation counts between macOS and Windows on every push, so drift is caught at the commit that introduces it.
- **Real-world import files vary beyond the one known sample (other clubs, other layouts), and over-fitting the row-skipping heuristics could silently drop real players** → The wizard always shows skipped rows explicitly with per-row include/exclude toggles and requires user confirmation before commit; import_run stores counts and decisions for audit; the fixture generator produces multiple structurally different messy files (varying header rows, delimiters, encodings) used in tests, and validation warnings (§8.6) surface anything ambiguous rather than deciding silently.
- **Spring Boot 3.5 is past OSS EOL at project start (final patch 3.5.16, June 2026)** → Acceptable short-term for a localhost-bound desktop backend with no exposed network surface and token-gated API; revisit when Timefold releases a 2.x-compatible path we can adopt (which also unlocks Boot 4). Keep controller/service code free of Boot-3-only idioms to keep the upgrade mechanical.
- **Explanation quality (the 'why not group C with his friend' answer) depends on what-if alternative comparisons that cost extra ScoreAnalysis evaluations and could be slow for 130 players × 12 groups** → Compute full alternative comparisons lazily per clicked participant (top-3 candidate groups only) rather than for all players at solve end; persist results in explanation_record as a cache; ScoreAnalysis on a ~130-entity problem is milliseconds-scale, and the M7 acceptance test includes a latency budget (<2s per explanation).
- **SQLite + Hibernate community dialect has rough edges (no ALTER COLUMN, single-writer, limited DDL in Flyway)** → Schema changes only via Flyway using the create-copy-rename idiom (documented rule for coding agents); single-user desktop app means no write concurrency; WAL mode + foreign_keys pragma set on the datasource; repository integration tests run against real SQLite files, not H2, so dialect issues surface in tests.

## 1. SQLite schema (spec §7)

**Migration approach:** Flyway `V1__core.sql` (persons→assignments), `V2__seed_constraints_and_standard_fields.sql`, `V3__runs_explanations_plans.sql`, `V4__import.sql`. Hibernate `ddl-auto=validate`. SQLite has no `ALTER COLUMN`; migrations that reshape tables use the create-copy-rename idiom — a rule written into `backend/CONTRIBUTING.md` for the coding agents.

**DB file location (spec §21.1):** resolved in `AppDataDirResolver.java`: macOS `~/Library/Application Support/Gruppindelning/gruppindelning.db`, Windows `%APPDATA%\Gruppindelning\gruppindelning.db`, overridable with `--app.data-dir=` (used by dev mode and tests; Tauri passes its `appDataDir`). JDBC URL `jdbc:sqlite:<path>?...` with `foreign_keys=on`, WAL mode.

## 2. Import pipeline (spec §8) — generic, wizard-driven

Server-side stateful `ImportSession` (in-memory map keyed by session id, holding the parsed workbook grid; nothing persisted until commit).

Wizard steps (frontend route `/plans/:planId/import`, backend drives each step):
1. **Välj fil** — `POST /api/import/sessions` (multipart xlsx/csv). POI `XSSFWorkbook` read with `DataFormatter` + Swedish locale; cell comments and threaded comments are simply ignored (POI tolerates them; regression test includes a fixture with both). CSV: BOM/charset sniff (UTF-8 → ISO-8859-1 fallback), delimiter sniff (`;` vs `,`, Swedish Excel exports use `;`). Every cell kept as `{raw: string(DataFormatter), type: STRING|NUMERIC|DATE|TIME|BLANK}` — Excel time values (`18:00:00`) survive as formatted text, no lossy coercion.
2. **Välj sheet** — `GET .../sheets`, pick one (e.g. Herr vs Dam ⇒ two separate imports into two ActivityPlans).
3. **Förhandsgranska + header-rad** — `GET .../preview?rows=30` returns the raw grid; backend suggests the header row (heuristic: first row among the top 10 maximizing count of distinct non-empty short text cells); user can override.
4. **Mappa kolumner** — each column maps to a target: `firstName, lastName, displayName, email, phone, rankingPoints, previousGroup, previousGroupLevel, manualLevelScore, comment→importedComment, customField:<key>, newCustomField, ignore` (spec §8.4). **No hardcoded column names**: suggestions come from (a) a matching `import_template` (exact `header_hash` of the header cell set), then (b) fuzzy similarity between header text and field labels/keys. Semi-structured columns like the real file's `Tid`/`RankInfo`/`Tränare` are mapped to text custom fields — imported as *reference* data only, never auto-interpreted (spec §2.2).
5. **Skapa saknade fält** — inline creation of `field_definition` rows for `newCustomField` targets (type + label; constraint wiring done later in Fältbyggaren).
6. **Validera** — `POST .../validate` classifies every data row: rows lacking any mapped name value are auto-marked **skip** (this generically absorbs the real file's mess: the stacked group-metadata column, blank separators, `Kölista` headers, count rows, refund-tracking rows — all lack first/last name). The UI shows included vs skipped rows with per-row toggle. Warnings per §8.6: missing name, duplicate rows, missing ranking, non-numeric numbers, unknown previous group, empty rows, and **potential person duplicates** against existing DB persons + within the file (exact email → exact phone → exact normalized name → Jaro-Winkler ≥ 0.92 on `förnamn efternamn`).
7. **Personmatchning** — for each potential match the user picks *Samma person* (link to existing `person`) or *Ny person* (spec §8.7); decisions stored in `import_run.decisions_json`.
8. **Importera** — `POST .../commit` in one transaction: upsert persons, create `participant_profile` per row (comment → `imported_comment`), write `custom_field_value`s, create `player_assignment(source=imported, group_id=NULL)`, record `import_run`, then offer "Spara som importmall" → `import_template`.

Test fixtures: a **generated** synthetic xlsx (`testdata/generator` Java class) reproducing the real file's shape — stacked metadata column, blank separators, mixed time cells, Swedish characters (å/ä/ö in names), cell + threaded comments, waiting-list section — with fake people. The real confidential file is never committed or read by tests.

## 3. Field builder & constraint config (spec §9)

- **Seeding:** migration `V2` inserts the 18 standard fields (§9.2) as global `field_definition` rows (`is_standard=1`), keys e.g. `rankingPoints(number, COLUMN)`, `previousGroup(text, COLUMN)`, `manualLevelScore(number, COLUMN)`, `canTimes/cannotTimes/preferTimes(timeRelation, CUSTOM)`, `playWith/mustPlayWith/avoidPlayWith(personRelation)`, `wantsCoach/mustHaveCoach/cannotHaveCoach(coachRelation)`, `newToClub(boolean)`, `needsManualReview(boolean, COLUMN→manual_review_flag)`, `priority(number)`, `importedComment/internalNote(text, INFO, affects_optimization=0, locked-off)`.
- **Field→constraint mapping:** `constraint_type` enum on the field: `LEVEL_BALANCE_INPUT, TIME_AVAILABILITY, TIME_PREFERENCE, SAME_GROUP, DIFFERENT_GROUP, COACH_PREFERENCE, COACH_FORBIDDEN, PRIORITY, NONE`. The Timefold constraint provider consumes *generic* problem facts (`PairingWish`, `TimeConstraintFact`, `CoachWishFact`) built by `SolverInputAssembler` from field values + field config — so a council-created custom personRelation field with `SAME_GROUP soft 80` behaves exactly like the standard one.
- **Weight flow:** effective weights = `constraint_definition.default_weight` ⊕ `constraint_weight_config` (per plan) ⊕ per-field `hard_or_soft/weight`. Assembler produces a `Map<String, HardSoftScore>` set on the planning solution via **`ConstraintWeightOverrides<HardSoftScore>`** (Timefold 1.x API); hard/soft toggle maps to `HardSoftScore.ofHard(w)` vs `ofSoft(w)`. The exact map used is snapshotted into `optimization_run.constraint_weights_json` so explanations always show the weights in force (spec §17).
- **Structured-entry UX (core flow):** Deltagarvyn row click opens a full-height drawer: **left pane** = `importedComment` + imported reference fields (`Tid`, `RankInfo`, …) read-only with a "Känslig — visas endast internt" badge; **right pane** = structured fields: `Kan inte tider`/`Föredrar tider` (checkbox group of the plan's TimeSlots), `Vill spela med`/`Måste spela med` (searchable picker over the plan's participants), `Vill ha tränare` (coach picker), `Manuell nivåscore` (number), `Behöver manuell bedömning` (switch), `Intern kommentar`. Keyboard next/previous participant for fast triage of ~130 rows. Saves via `PATCH /api/participants/{id}` + `PUT /api/participants/{id}/field-values`.

## 4. Resources & coaches (spec §12, §13)

- **Tider & banor view:** CRUD time slots (dag, start, slut, label). Per slot a `courtCount` spinner → `POST /api/plans/{id}/training-blocks/generate` creates/reuses courts `Bana 1..N` (default venue) and upserts `training_block` rows; blocks removed only if unassigned. Grid of blocks (slot × court) with active toggle (§12.3) and lock toggle.
- **Tränarvyn:** create coach from existing person or new; edit `coach_level`, `can_coach_min/max_level`, `max_groups_per_day/week`, `can_also_train_as_participant`, per-slot availability tri-state (available/unavailable/preferred → `coach_time_slot`), internal note. Badge when the coach is also a participant in an overlapping plan.

## 5. Capacity analysis (spec §12.4)

`CapacityService` → `GET /api/plans/{id}/capacity`: `participantCount, activeBlockCount, plannedGroupCount, targetSize, maxSize, targetCapacity = blocks×target, maxCapacity = blocks×max, waitlistRisk (participantCount − maxCapacity), coachCount, groupsRequiringCoach, coachShortage per slot (available coaches vs blocks in slot), timeConstraintConflicts (participants whose cannotTimes exclude every slot), statusText` with levels OK/VARNING/KRITISK. Shown in Kapacitetsvyn and as a pre-solve gate banner in Optimeringsvyn.

## 7. Export (spec §20, §21.3)

`ExportService` (POI + Commons CSV), `POST /api/plans/{id}/export` body `{format: xlsx|csv, includeComments: false (default), layout: grouped|flat}`:
- **xlsx grouped** (mirrors the council's current presentation): one sheet per plan; per group a header block (`Gruppnamn`, `Tid`, `Bana`, `Tränare`) then player rows (`Namn, Ranking, Nivåscore, Tidigare grupp, Varningar`), player count row, blank separator; a `Kölista` section for unassigned/waitlisted participants.
- **csv flat**: one row per player with `Grupp, Tid, Bana, Tränare, ...` columns; `;`-delimited UTF-8 with BOM (opens correctly in Swedish Excel).
- `includeComments=true` must be explicitly ticked ("Inkludera kommentarer i export" default OFF, §20.3); adds `Kommentar`/`Intern kommentar` columns.
- **Anonymized test dataset** `POST /api/plans/{id}/export/anonymized` → JSON + xlsx with names→`Spelare 001`, email/phone/comments/externalId stripped; ranking, levels, structured constraints, group sizes, slots, courts preserved (spec §21.3) — the format the solver regression fixtures use.

Solve runs via Timefold `SolverManager` (one job at a time), config: `HardSoftScore`, `environmentMode` reproducible default (`PHASE_ASSERT`), `<randomSeed>0</randomSeed>`, deterministic input sort by entity id before solve (§16.5, §4.5). Explanations: every constraint uses `justifyWith(...)` producing typed justifications carrying participant/group ids; after solve `SolutionManager.analyze(solution, FETCH_ALL)` → `ExplanationBuilder` groups matches per participant into positive/negative factors, computes alternative-group comparisons via what-if `analyze` diffs for top-3 candidate groups, persists `explanation_record`. What-if move = clone last solution, apply move, `analyze` both, `ScoreAnalysis.diff` + direct group-size/level-spread deltas.
