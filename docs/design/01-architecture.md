# Design: Architecture, Packaging, Repo & Build

> Generated by design agents 2026-07-01, web-verified. 

## Executive summary

The design follows the spec's preferred stack — Tauri v2 shell + React/Vite/TypeScript frontend + a Spring Boot/Timefold Java backend over localhost REST with SQLite — with one decisive, web-verified correction: Timefold Solver 2.x (April 2026) moved the score-analysis/explainability APIs into a paid edition, so the project must pin the last fully open-source line, Timefold 1.33.0 (Apache-2.0), which still includes SolutionManager.analyze() and the recommendation API that the spec's core explainability requirements depend on. That pin in turn fixes the backend at Spring Boot 3.5.x (Timefold 1.x has no Boot 4 support) with the official starter, sqlite-jdbc + Flyway + JdbcClient (no JPA), and Temurin 21 bundled as a jlink custom runtime shipped inside Tauri's bundle.resources alongside the fat jar — spawned by ~200 lines of Rust implementing a deterministic handshake (bind 127.0.0.1, --server.port=0, random session token, GP_READY stdout line, health polling, graceful shutdown plus a parent-PID watchdog). The monorepo lives in an app/ subdirectory beneath the confidential folder so the real xlsx is physically outside the repo, backed by deny-by-default spreadsheet ignores and a personnummer-scanning pre-commit/CI guard. Development runs the frontend as a plain browser SPA against a fixed-port dev backend (all file I/O flows through multipart upload/download, Tauri APIs isolated in one platform.ts module), so Sonnet coding agents and Playwright can verify every increment without the desktop shell. CI runs a macOS+Windows matrix that asserts exact golden solver scores from committed anonymized datasets (REPRODUCIBLE mode, fixed seed, integer-scaled HardSoftLongScore), making the spec's cross-platform determinism requirement a build-breaking gate, plus a smoke job that boots the jlinked runtime with the real jar. Releases produce ad-hoc-signed .dmg (arm64 + x64, cross-targeted jlink) and NSIS .exe with documented Gatekeeper/SmartScreen workarounds for the unsigned nonprofit distribution, and clear paid signing upgrade paths. Frontend data layer is TanStack Query v5 + Zustand with AG Grid Community and Mantine, deliberately pinned one major behind the newest releases (Vite 7, Mantine 8, AG Grid 35) because agent reliability on well-trained APIs outweighs novelty for a review-gated implementation pipeline.

## Key decisions

- **Pin Timefold Solver 1.33.0 (last Apache-2.0 line with explainability) and never upgrade to 2.x**
  - Rationale: Verified July 2026: Timefold 2.0 (April 2026) moved SolutionManager.analyze()/ScoreAnalysis and the recommendation API into the paid 'Timefold Solver Plus' edition. Explainability is a core spec requirement (§17, §23.7); 1.33.0 is the latest Community release with these APIs under Apache-2.0. Also constrains framework choice since 1.x supports Spring Boot 3.x / Java 17+, not Boot 4.
  - Rejected: Timefold 2.2.x (current): explainability requires paid license; also raises Java baseline to 21 and removes 1.x APIs agents would rely on. OptaPlanner: unmaintained predecessor.
- **Tauri v2 (2.11.x) as desktop shell, with the Java backend bundled as resources and spawned from ~200 lines of Rust**
  - Rationale: Spec prefers Tauri; frontend is pure browser code so Electron's Node main process adds nothing; Tauri produces .dmg/NSIS .exe installers natively; ~10-15 MB shell vs ~100 MB Chromium; WebView2 preinstalled on Win10/11 with downloadBootstrapper fallback. Rust surface is fenced into one file (backend.rs).
  - Rejected: Electron: two bundled runtimes (Node+JRE), much larger, more signing friction; kept as documented fallback since the frontend is shell-agnostic.
- **Spring Boot 3.5.x (pinned) + official timefold-solver-spring-boot-starter 1.33.0, fat jar, Maven with wrapper**
  - Rationale: AI-agent friendliness is the top criterion: Spring Boot 3.x is the best-represented Java framework in training data; Timefold ships an official 1.x starter for Boot 3.x; 2-3 s startup is hidden behind a splash/health-check. Boot 3.5 OSS support ended 2026-06-30, accepted because Timefold 1.x doesn't support Boot 4 and the server binds 127.0.0.1 with token auth only (ADR documents the post-MVP Boot 4 path: drop the starter, programmatic SolverFactory).
  - Rejected: Spring Boot 4.1 (current): no Timefold 1.x starter support, thinner agent training data. Quarkus 3.x: faster startup but build-time DI error modes agents handle poorly. Javalin/plain: too much hand-rolled glue (DI, multipart, OpenAPI, tests) for a review-gated agent pipeline.
- **SQLite via xerial sqlite-jdbc 3.53.x + Flyway migrations + Spring JdbcClient; no JPA/Hibernate**
  - Rationale: Hibernate's SQLite dialect is community-tier and fights SQLite's limited ALTER; explicit SQL migrations + JdbcClient row mappers are deterministic, reviewable, and reliably generated by agents. WAL mode + busy_timeout pragmas for robustness.
  - Rejected: Spring Data JPA + hibernate-community-dialects: dialect quirks and schema-evolution pain on SQLite outweigh CRUD convenience.
- **Bundle Temurin 21 LTS as a jlink custom runtime shipped in Tauri bundle.resources (with the fat jar); not jpackage, not externalBin sidecar**
  - Rationale: Tauri externalBin requires a single file per target triple; jlink output is a directory, so resources is the correct mechanism. jpackage would only add a redundant launcher/installer stage. Curated module list (~45-60 MB), per-target images for macOS arm64/x64 and Windows x64; jlink cross-targets via target-platform jmods so mac-x64 builds on arm runners. Java 21 is the intersection certified by Timefold 1.x and Boot 3.5 (Java 25 LTS is not certified for Boot 3.5). CI smoke test launches the jlinked runtime and curls /api/health to catch missing modules.
  - Rejected: jpackage app-image (redundant launcher, conflicts with Tauri installers); externalBin sidecar (directory payloads unsupported); requiring system Java (spec §4.3 forbids).
- **Localhost REST contract: 127.0.0.1 + --server.port=0, random per-session token, stdout GP_READY handshake, health polling, POST /shutdown + parent-PID watchdog**
  - Rationale: port=0 eliminates port-in-use failures by construction; loopback binding avoids Windows Firewall prompts and network exposure; token header blocks other local processes; stdout READY line + health poll gives a deterministic 30 s-bounded startup; watchdog thread in the backend guarantees no orphaned java.exe even if the shell crashes; crash mid-session surfaces via child-exit event + TanStack Query health poll with a restart flow.
  - Rejected: Fixed port (collision-prone); port-file-only discovery (race/staleness; kept only as log artifact); Tauri IPC-only (would force all file/db logic into Rust).
- **Git repo lives in the app/ subdirectory of the confidential folder; deny-by-default .gitignore for all spreadsheets + committed-CSV allowlist + pre-commit/CI confidentiality scanner; xlsx test fixtures generated at test time from anonymized CSVs**
  - Rationale: Physical separation (repo root below the confidential files) means the real xlsx can never be staged; layered guards (ignore rules, personnummer-regex scanner in pre-commit and CI, CLAUDE.md agent rules) satisfy spec §26.15 and protect against agent mistakes.
  - Rejected: Repo at folder root with .gitignore only: one 'git add -f' or ignore-file edit away from leaking personal data.
- **Frontend: React 19 + TypeScript 5 + Vite 7 (pinned, not Vite 8), TanStack Query v5 + Zustand 5, AG Grid Community 35.x (not week-old v36), Mantine 8.x (not 9.4), hand-rolled CSS-grid schedule view, OpenAPI-generated TS types, Vitest + Playwright**
  - Rationale: Data-dense internal tool: AG Grid gives inline editing/sorting/filtering out of the box for the participants table; Mantine covers forms/modals/AppShell. Deliberately pin one major behind the newest releases (Vite 8 Rolldown swap, AG Grid 36, Mantine 9 are all <3 months old) because Sonnet coding agents are far more reliable on the previous majors; upgrades are post-MVP ADRs. springdoc OpenAPI -> openapi-typescript gives a single typed contract both agent teams compile against.
  - Rejected: TanStack Table (headless: agents re-implement editing plumbing); shadcn/ui (more assembly per component); Redux (overkill); calendar libraries for the 4x4 schedule grid (bespoke CSS grid is smaller).
- **Dev workflow: frontend must run fully in a plain browser (Vite 5173 -> proxy -> Boot dev on fixed port 4517); file import is multipart upload, export is byte download; Tauri APIs isolated behind src/lib/platform.ts; tauri dev skips backend spawn**
  - Rationale: Coding agents and Playwright can build/verify every feature without a running desktop shell, keeping increments cheaply testable in CI on Linux; the shell integration stays a thin, separately-tested layer.
  - Rejected: Tauri fs/dialog-centric file handling (breaks browser dev and e2e); Electron-style everything-through-IPC.
- **CI: mac+win matrix asserting exact golden scores (expected-scores.json) for three committed anonymized datasets in REPRODUCIBLE mode with fixed seed and integer-scaled HardSoftLongScore**
  - Rationale: Asserting both OSes against one committed golden file makes cross-platform divergence (spec §22.3) a CI failure by construction, stronger than diffing artifacts between jobs; integer score arithmetic removes the main floating-point nondeterminism source; desktop smoke job validates the jlinked runtime + jar on both OSes; release workflow ships aarch64+x64 .dmg (ad-hoc signed, signingIdentity '-') and NSIS .exe with unsigned-app install docs.
  - Rejected: Score-diffing between parallel CI jobs (weaker, flaky orchestration); double/BigDecimal-weighted scores (FP divergence risk); signing in MVP (no certificates available; documented Azure Trusted Signing / SignPath / Apple Developer paths for later).

## Risks

- **Timefold 1.33.0 is a frozen, maintenance-only line: no bug fixes or Java-version support beyond what exists today, and its docs are no longer the 'latest' docs online, so coding agents may accidentally write 2.x-style APIs.** → Pin 1.33.0 everywhere with maven-enforcer; CLAUDE.md hard-links the 1.33.x documentation URL and forbids docs.timefold.ai/latest; ConstraintVerifier tests catch API misuse at compile/test time; Apache-2.0 permits forking/patching if a blocking bug appears; nonprofit licensing of Solver Plus is a documented fallback if score-analysis needs ever outgrow 1.x.
- **Unsigned binaries: macOS 15 Gatekeeper requires System Settings 'Open Anyway' and Windows SmartScreen warns; antivirus may sandbox or delay the spawned bundled java, causing startup failures for council members.** → Ad-hoc codesign (signingIdentity '-' + deep codesign of the embedded JRE) so Apple Silicon doesn't kill the binary; illustrated install guides per OS linked from every release; 30 s handshake timeout surfaces a friendly error with log tail and retry; budget path to SignPath/Azure Trusted Signing (Windows) and Apple Developer nonprofit fee waiver documented; NSIS per-user install avoids admin/UAC friction.
- **Spring Boot 3.5's OSS support window ended 2026-06-30, so the pinned backend framework receives no further open-source patches during development.** → Accepted for a loopback-only, token-authenticated desktop process with no network exposure; ADR records the upgrade path (drop the Timefold starter, configure SolverFactory programmatically, move to Boot 4.x post-MVP); dependabot monitors CVEs on the actual attack-surface libs (POI, sqlite-jdbc, jackson).
- **jlink module list misses a reflectively-loaded JDK module (POI/AWT, sqlite-jdbc native loading, Spring instrumentation), producing a packaged app that crashes only on end-user machines.** → Curated explicit module list (includes java.desktop, java.xml, jdk.unsupported, java.instrument, jdk.crypto.ec) rather than raw jdeps output; a dedicated CI smoke job on mac and windows launches the jlinked runtime with the real fat jar and asserts /api/health plus one import+solve round-trip before any release artifact is produced.
- **Cross-platform score divergence (spec §22.3) from floating point, locale, encoding, or iteration-order differences between macOS and Windows.** → Integer-scaled HardSoftLongScore only (no double weights); environmentMode=REPRODUCIBLE + fixed randomSeed; deterministic sort of all facts/entities by stable id pre-solve; forced -Duser.timezone=UTC -Dfile.encoding=UTF-8 on the spawned JVM; identical bundled Temurin build on both OSes; golden expected-scores.json asserted on the mac+win CI matrix on every PR so any divergence is caught immediately.
- **Confidential data leak: the real xlsx sits one directory above the repo, and autonomous coding agents create/commit files.** → Repo rooted in app/ subdirectory (physical separation); deny-by-default ignore rules for all spreadsheet/CSV files with a narrow committed-dataset allowlist; pre-commit + CI scanner for spreadsheet extensions, personnummer regex, and known real-file names; CLAUDE.md prohibition on reading the real file; all xlsx test fixtures generated at test runtime from anonymized CSVs.
- **Orphaned backend processes or port/token races if the shell crashes, is force-quit, or two instances start.** → Parent-PID watchdog thread in the backend self-terminates within 5 s of shell death; graceful POST /shutdown with 3 s kill fallback on normal exit; --server.port=0 eliminates port collisions; per-session random token; tauri-plugin-single-instance prevents double launch.
- **Deliberately pinning one-major-behind frontend deps (Vite 7, Mantine 8, AG Grid 35) accrues upgrade debt and may draw reviewer objections.** → Each pin is an ADR with the agent-reliability rationale and a revisit-after-MVP trigger; all pins are exact versions in lockfiles so upgrades are isolated, reviewable PRs; none of the pinned lines are EOL or unpatched for security at time of writing.

# Architecture, Packaging, Repo Layout & Build/Dev Workflow — Design Document

Scope: desktop shell, backend framework, Java runtime bundling, frontend–backend contract, repo layout, dev workflow, packaging/CI, frontend stack. Aligned with `utkast-kravspec.txt` (§4 arkitektur, §4.3 bundled JRE, §4.5 + §22.3 cross-platform determinism, §21 local storage, §26 agent instructions).

---

## 0. The single most important dependency decision: Timefold 1.x, not 2.x

**Verified via web (July 2026):** Timefold Solver 2.0 shipped April 2026 and **moved the explainability APIs (`SolutionManager.analyze()` / ScoreAnalysis, and the recommendation API) out of the Apache-2.0 Community Edition into the paid "Timefold Solver Plus" edition**. Explainability is a *core requirement* of this spec (§17, §23.7). The last fully open-source line with score analysis is **1.x, latest release 1.33.0** (maintenance release; Apache-2.0; ScoreAnalysis available since Community 1.4.0; `recommendAssignment`/RecommendedFit API also Community in 1.x). Timefold 2.x also raised the Java baseline to 21 and its "latest" docs now describe 2.x APIs.

**Decision: pin `ai.timefold.solver:timefold-solver-core:1.33.0` (+ `timefold-solver-spring-boot-starter:1.33.0`, `timefold-solver-test:1.33.0`) and never float this version.** Record in `CLAUDE.md` that coding agents must use the 1.33 docs URL (`https://docs.timefold.ai/timefold-solver/1.33.x/...`), because `docs.timefold.ai/timefold-solver/latest` now documents 2.x and will mislead them (e.g. 2.x removed `ConstraintConfiguration` patterns and renamed APIs).

This also fixes the framework question’s constraint set: Timefold 1.x supports Java 17+, Spring Boot 3.x and Quarkus 3.x — **not** Spring Boot 4.

## 1. Desktop shell: Tauri v2 (chosen) vs Electron (rejected)

**Decision: Tauri v2 (CLI/core 2.11.x, verified current stable July 2026).**

Rationale:
- The spec explicitly prefers Tauri (§4.2). The frontend is plain React+Vite; we need no Node runtime in production because *all* file parsing, SQLite and solving happen in the Java backend. Electron's main advantage (Node in main process) buys us nothing here.
- Bundle size/footprint: Tauri app ≈ 10–15 MB + our ~90 MB JRE/jar payload; Electron adds ~100 MB Chromium per platform on top of the same Java payload.
- Installers out of the box: `.dmg`/`.app` (macOS), NSIS `.exe` and WiX `.msi` (Windows) via `tauri build` — matches §4.3 exactly.
- Sidecar/resource support: Tauri v2 has first-class bundling of extra binaries/resources and a maintained shell/process API (`tauri-plugin-shell`, verified in v2 docs).
- Rust knowledge needed is minimal and fenced: ~200 lines in one file (`backend.rs`) for spawn/handshake/shutdown. Everything else is web code. This keeps the AI-agent surface mainstream.

Electron rejected because: heavier, requires bundling *two* runtimes (Node + JRE), electron-builder signing/notarization defaults create more friction for an unsigned nonprofit app, and the spec prefers Tauri. It remains the documented fallback if Tauri's WebView causes an unforeseen blocker (the escape hatch is cheap: frontend is pure browser code, see §6.2 below).

**WebView2 on Windows:** Tauri uses the Evergreen WebView2 runtime, preinstalled on Windows 10/11. `tauri.conf.json` → `bundle.windows.webviewInstallMode: { "type": "downloadBootstrapper" }` so the rare machine without it gets it during install. No action needed from users on normal machines.

**Unsigned-app reality (no Apple Developer account, no Windows cert):**
- macOS: an unsigned, un-notarized `.dmg` download gets the quarantine attribute. On macOS 15+ (Sequoia) right-click→Open alone no longer suffices; the user must go to **System Settings → Privacy & Security → "Open Anyway"** once. On Apple Silicon a binary must have at least an *ad-hoc* signature or it is killed — set `bundle.macOS.signingIdentity: "-"` in `tauri.conf.json` (ad-hoc signing, no certificate needed) and add a CI step `codesign --force --deep --sign - GroupPlanner.app` after injecting the JRE so the embedded `java` binary is also ad-hoc signed. Ship a one-page illustrated `docs/installation-mac.md` and link it next to every release download.
- Windows: SmartScreen shows "Windows protected your PC" → **More info → Run anyway**. Document with a screenshot in `docs/installation-windows.md`. Prefer the NSIS `.exe` (per-user install under `%LOCALAPPDATA%`, no admin prompt) as the primary artifact; `.msi` optional.
- Mitigation path if friction is too high: (a) Azure Trusted Signing (low monthly cost, org validation) or SignPath.io free OSS signing for Windows; (b) Apple Developer Program ($99/yr; Apple offers fee waivers for eligible nonprofits) for Developer ID + notarization. Neither blocks MVP; both slot into the existing CI signing hooks (`APPLE_SIGNING_IDENTITY`, tauri `signCommand`).

## 2. Backend framework: Spring Boot 3.5.x (chosen) vs Quarkus vs Javalin

**Decision: Spring Boot 3.5.x (latest 3.5 patch, pinned) with `timefold-solver-spring-boot-starter:1.33.0`, `spring-boot-starter-web`, `spring-boot-starter-jdbc`, packaged as a repackaged fat jar. Maven (already installed; use committed `mvnw` wrapper pinned to 3.9.x).**

Rationale:
- **AI-agent friendliness is the top-weighted criterion** per the implementation model (Sonnet agents in small reviewed increments). Spring Boot 3.x is the single best-represented Java framework in training data; controllers, `JdbcClient`, `@SpringBootTest`, MockMvc are patterns agents produce correctly on the first try.
- Timefold ships an official Spring Boot starter for 1.x (verified: `ai.timefold.solver:timefold-solver-spring-boot-starter` on Maven Central; Timefold 1.x supports Spring Boot 3.x, Java 17+). We get injectable `SolverManager`/`SolutionManager` and property-based solver config (`environmentMode=REPRODUCIBLE`, seed, termination) for free.
- Startup time ~2–3 s for web+jdbc on modern hardware is fine for a desktop app: the shell shows the UI immediately with a "Startar motorn…" splash state while health-checking (see §4).
- **Version caveat (verified):** Spring Boot 4.1 is the current line (4.1.0 released 2026-06-10), and Boot 3.5's OSS support window ended 2026-06-30. We still pin 3.5.x because (a) Timefold 1.x does **not** support Boot 4 (Boot 4 support arrived in the Timefold 2.x era), and (b) the backend binds to 127.0.0.1 only, is token-authenticated, and processes only local files — the attack surface of a slightly-stale framework is negligible here. Documented as an ADR; post-MVP option: drop the starter and configure `SolverFactory` programmatically (~30 lines) to unlock Boot 4.

Rejected:
- **Quarkus 3.x**: faster startup (~1 s) and an official Timefold extension, but build-time DI/bytecode transformation produces error modes AI agents handle poorly, and its test idioms are less common in training data. Startup delta (~1–2 s) is not user-noticeable behind a splash.
- **Javalin/plain Java**: smallest and fastest, but agents must hand-roll DI, config, validation, multipart handling, OpenAPI and test harness — more bespoke glue is exactly what a review-gated agent pipeline should avoid.

**Persistence: SQLite via `org.xerial:sqlite-jdbc:3.53.x` (verified current 3.53.2.0) + Flyway migrations + Spring `JdbcClient` (no JPA/Hibernate).** Hibernate's SQLite dialect is community-tier and its DDL quirks (no real `ALTER COLUMN`) fight migrations; explicit SQL in Flyway files + `JdbcClient` row mappers is deterministic, trivially reviewable, and agents write it reliably. DB file lives in the OS app-data dir passed in by the shell (spec §21.1). Set pragmas on the datasource: `journal_mode=WAL`, `foreign_keys=ON`, `busy_timeout=5000`.

**Excel/CSV:** Apache POI 5.5.x (verified current 5.5.1) for `.xlsx` (must tolerate cell comments/threaded comments per the sample-file structure), Commons CSV for `.csv`.

**API docs/typing:** `springdoc-openapi-starter-webmvc-api` (2.x line for Boot 3) exposes `/v3/api-docs`; frontend runs `openapi-typescript` in a `npm run typegen` script → `frontend/src/api/schema.d.ts`. One source of truth for the contract; agents on both sides compile against it.

**Determinism (spec §4.5/§16.5/§22.3):** use **integer-scaled `HardSoftLongScore`** (never floating-point weights), `environmentMode=REPRODUCIBLE`, fixed `randomSeed`, sort all problem facts/entities by stable id before solve, and force `-Duser.timezone=UTC -Dfile.encoding=UTF-8` on the spawned JVM. Same Temurin build on both OSes (bundled), all Maven versions pinned (no ranges), `maven-enforcer-plugin` with `requireUpperBoundDeps`.

## 3. Java runtime bundling

**Decision: jlink custom runtime (Temurin 21 LTS, latest 21.0.x patch) shipped as a Tauri *resource* directory next to the fat jar — not jpackage, not Tauri `externalBin`.**

- Tauri's `externalBin` (sidecar) requires a *single file* per target triple; both a jlink runtime and a jpackage app-image are directories. So the correct Tauri mechanism is `bundle.resources`. jpackage would only add a native launcher we don't need (the Rust shell is the launcher) and its own installer stage that would conflict with Tauri's.
- Java version: **Temurin 21** (LTS). Timefold 1.33 requires 17+; Boot 3.5 supports 17–24; Java 25 LTS exists but is not certified for Boot 3.5 — 21 is the intersection everyone tests against. Same Temurin patch version bundled on all platforms (spec §4.5.1).
- Build (in `scripts/build-jre.sh` / `build-jre.ps1`):
  1. `jdeps --ignore-missing-deps --print-module-deps --multi-release 21 backend.jar` as a starting point, but pin the module list explicitly in the script (jdeps misses reflective deps): `java.base,java.sql,java.naming,java.desktop,java.management,java.instrument,java.net.http,java.security.jgss,java.xml,jdk.unsupported,jdk.crypto.ec` (POI needs `java.desktop`+`java.xml`; sqlite-jdbc needs `jdk.unsupported`; Spring needs `java.instrument`).
  2. `jlink --module-path $JMODS --add-modules $MODULES --strip-debug --no-header-files --no-man-pages --compress zip-6 --output desktop/src-tauri/resources/jre` → ~45–60 MB.
- **Per-platform targets:** macOS arm64, macOS x64, Windows x64. jlink can cross-target: pass `--module-path` pointing at the *target platform's* downloaded Temurin `jmods`. So the mac x64 build can run on an arm64 runner (download Temurin 21 macos-x64 jmods, jlink, `tauri build --target x86_64-apple-darwin`). CI keeps it simpler by using native runners where available (see §7).
- A packaged-runtime **smoke test** in CI launches `resources/jre/bin/java -jar backend.jar` and asserts `/api/health` returns 200, so a missing jlink module can never reach a release.

## 4. Frontend–backend contract: localhost REST + handshake protocol

Bind **127.0.0.1 only** (`--server.address=127.0.0.1`) — no LAN exposure, and loopback binding avoids the Windows Firewall dialog. Random free port via `--server.port=0`. All requests carry a per-session random token (defense against other local processes).

**Handshake protocol (implemented in `desktop/src-tauri/src/backend.rs`):**
1. Shell generates `token` (32 random hex bytes) and resolves `dataDir` (Tauri `app_data_dir()` → `~/Library/Application Support/GroupPlanner/` on mac, `%APPDATA%\GroupPlanner\` on win — spec §21.1).
2. Spawn with `std::process::Command`: `{resources}/jre/bin/java -Duser.timezone=UTC -Dfile.encoding=UTF-8 -jar {resources}/backend/backend.jar --server.port=0 --server.address=127.0.0.1`, env `GP_TOKEN`, `GP_DATA_DIR`, `GP_PARENT_PID`; stdout/stderr piped; on Windows use `CREATE_NO_WINDOW` (0x08000000) so no console flashes.
3. Backend, on `WebServerInitializedEvent`: logs to `{dataDir}/logs/backend.log` (Logback rolling) **and** prints exactly one line to stdout: `GP_READY {"port":54321,"pid":4242,"schemaVersion":3}`. A servlet filter rejects any request whose `X-GP-Token` header ≠ `GP_TOKEN` with 401.
4. Shell reads stdout lines with a **30 s timeout**; on `GP_READY` it polls `GET /api/health` (every 500 ms, ≤10 s) until `{"status":"UP"}`. Then it stores `{base_url, token}` in Tauri state and exposes command `get_backend_info` to the webview; the frontend fetch wrapper reads it once and attaches the header.
5. **Shutdown:** on `RunEvent::ExitRequested` → `POST /api/system/shutdown` (backend calls `SpringApplication.exit`), wait ≤3 s, then hard-kill the child if still alive. Belt-and-braces: backend watchdog thread checks `ProcessHandle.of(GP_PARENT_PID)` every 5 s and self-terminates if the shell died (covers shell crash / force-quit).

**Failure modes:**
- *Port in use*: impossible by construction (`port=0`).
- *Backend exits before READY / timeout*: shell emits `backend-failed` event → frontend shows an error screen with the last 50 log lines and the path to `backend.log`, plus a "Försök igen" (respawn) button. Typical cause: antivirus sandboxing the unsigned `java` — the error page links the install docs (Defender "Run anyway"/exclusion instructions).
- *Backend crash mid-session*: shell watches the child; on unexpected exit it emits `backend-crashed`. Frontend also runs a TanStack Query health poll (`refetchInterval: 5000` on `/api/health`) and flips to a reconnect overlay; restart re-runs the full handshake (new port + token) and invalidates all queries. SQLite in WAL mode means no data loss for committed work.
- *Two app instances*: `tauri-plugin-single-instance` focuses the existing window instead of spawning a second backend.

## 6. Dev workflow

Prereqs (dev machine, one-time): `brew install --cask temurin@21`, `brew install rustup && rustup-init -y` (stable toolchain; Tauri v2 needs recent stable Rust), Node 24 LTS via `brew install node@24` (Node 23 currently installed is a dead non-LTS line — switch; verified Node 24 = Active LTS July 2026). Root `package.json` scripts (npm workspaces + `concurrently`):

```jsonc
{
  "scripts": {
    "dev:backend":  "cd backend && ./mvnw spring-boot:run -Dspring-boot.run.profiles=dev",   // fixed port 4517, dev token "dev", CORS for :5173
    "dev:web":      "npm --workspace frontend run dev",                                      // Vite on 5173, proxy /api -> 127.0.0.1:4517
    "dev":          "concurrently -k \"npm:dev:backend\" \"npm:dev:web\"",                   // ← 90% of development: plain browser
    "dev:desktop":  "npm --workspace desktop run tauri dev",                                 // real shell; skips backend spawn in dev, uses 4517
    "typegen":      "npm --workspace frontend run typegen",                                  // openapi-typescript from running backend

    "test:web":     "npm --workspace frontend run test",                                     // vitest

    "package":      "bash scripts/package.sh"                                               // local production build for current platform
  }
}
```

**Key workflow decision: the frontend must be fully functional in an ordinary browser against the dev backend.** File import is a multipart upload (`POST /api/import/files`) — identical in browser and webview; export downloads bytes (anchor download in browser, `plugin-dialog` save in Tauri via `platform.ts`). This means coding agents and Playwright never need a running Tauri shell for feature work, which makes every increment cheaply verifiable. Hot reload: Vite HMR for UI; Spring DevTools restart for backend; `tauri dev` only when touching the shell.

## 7. Packaging pipeline & GitHub Actions CI

**`scripts/package.sh` (per platform/arch, also what CI runs):**
1. `cd backend && ./mvnw -DskipTests package` → `backend/target/backend.jar` (spring-boot repackage, `finalName=backend`).
2. `scripts/build-jre.sh <temurin-jmods-for-target>` → `desktop/src-tauri/resources/jre/`.
3. `cp backend/target/backend.jar desktop/src-tauri/resources/backend/backend.jar`.
4. `npm --workspace frontend run build` → `frontend/dist` (tauri.conf `frontendDist: "../../frontend/dist"`).
5. `npm --workspace desktop run tauri build [--target x86_64-apple-darwin]` → `.dmg`/`.app` or NSIS `.exe`; on macOS follow with the ad-hoc `codesign --force --deep --sign -` step.

**`.github/workflows/ci.yml` (every PR):**
- `guard`: `scripts/check-no-confidential.sh` over the full tree (fail-fast).
- `backend`: matrix `[macos-latest, windows-latest]`, `actions/setup-java` (Temurin 21), `./mvnw verify`. This *includes* `SolverRegressionTest`, which solves the three committed datasets (10/20/120 players, spec §22.2) in REPRODUCIBLE mode with the fixed seed and asserts **exact equality** with `test-data/regression/expected-scores.json` (score + hard-violation count + assignment hash). Because the same golden file is asserted on both runners, mac/win divergence fails CI by construction — this is the spec §22.3 gate, and it's stronger than artifact-diffing between jobs.
- `frontend`: ubuntu — `npm ci`, typecheck (`tsc --noEmit`), ESLint, Vitest; then boot `backend.jar` and run Playwright e2e against `vite preview` (import→map→optimize→explain→export happy path, spec §22.4).
- `desktop-smoke` (matrix mac/win): run steps 1–3 of package.sh, then launch `resources/jre/bin/java -jar resources/backend/backend.jar` and curl `/api/health` — catches jlink module misses without a full bundle.

**`.github/workflows/release.yml` (on tag `v*`):** matrix builds → artifacts:
- `macos-latest` (arm64): `.dmg` aarch64; plus x64 `.dmg` via `rustup target add x86_64-apple-darwin` + cross-target jlink (x64 jmods) — avoids dependence on Intel runners.
- `windows-latest`: NSIS `.exe` (primary), `.msi` optional.
- Upload to a GitHub Release with install-instruction links (unsigned-app guidance) in the release notes. `tauri-action` may be used, but plain `npm run tauri build` + `softprops/action-gh-release` keeps it transparent.
