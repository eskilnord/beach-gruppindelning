# Solver Design Verification (API re-verification + soundness)


## Solver verifier

### Verdict

APPROVE WITH FIXES — the document is implementation-grade on the Timefold 1.33.0 API surface: of ~21 independently re-verified claims (against the v1.33.0 source tag), everything load-bearing is correct — allowsUnassigned, @PlanningPin entity-level semantics, ConstraintWeightOverrides (incl. zero-weight-vanishes and level reclassification), HardMediumSoftLongScore, analyze/FETCH_ALL/diff(this−other), recommendAssignment in Community, ConstraintVerifier incl. justifiesWith, TerminationConfig keys, EnvironmentMode (PHASE_ASSERT default, NO_ASSERT prod), loadBalance-returns-BigDecimal, the Enterprise feature boundary, and the Spring starter wiring. Only two API corrections are needed: replace the deprecated withBestSolutionConsumer/withFinalBestSolutionConsumer with the *EventConsumer variants, and fix §9.3's static-looking SolverConfigOverride call (instance method). The entity-split + @PlanningPin lock design genuinely satisfies §15.2/§15.3 per-aspect independence, integer scaling has >1e8 headroom with consistent floorDiv rounding, GroupOrderByLevel is expressible as specified, and the medium-level waitlist can never trade against hard constraints nor be bought off by soft quality — by default. Three majors must be fixed before this becomes the contract: (1) the waitlist explanation's 'placeable but score worsens / lower priority' branch is impossible under lexicographic scoring — priority narratives must come from data/swap comparison, not the single-move probe; (2) the waitlist invariant is only as strong as the weight-override guardrails, which don't exist yet (disabling unassignedPlayer or reclassifying wishes to MEDIUM silently breaks §2's guarantees); (3) the PinnedOverflowTest contract row asserts behavior the solver will correctly not exhibit. Remaining findings are minors: empty-group blindness in grouped-stat constraints, candidate-set cost arithmetic (12 probes not 5), the waitlisted-friend edge of the canonical question, greedy-baseline tie-breaks, CoachSlot id collision, complement-key weight fan-out, class-pinning-before-first-solve ordering, and an overloaded M-S1 that should shed LevelService/GreedyBaseline to M-S2 and name its Flyway schema dependencies.

### Problems

- **[MAJOR]** §2.3/§11.3 waitlist explanation branch 4 ('plats fanns men score försämras med X — lägre prioritet än de placerade') is mathematically impossible under lexicographic HardMediumSoftLongScore. Moving an unassigned player into any hard-feasible group ALWAYS improves the score: medium improves by 100×priority and medium dominates any soft worsening. So at a single-move-optimal solution, every candidate group for a waitlisted player is hard-blocked; if a probe ever returns 'placeable but worse', it actually means the solver terminated before local optimality — and the priority-comparison narrative is never derivable from the single-move what-if probe. The verdict enum {WOULD_BREAK_HARD, WORSE, BETTER} collapses to {WOULD_BREAK_HARD, BETTER} for unassigned players.
  - **Fix:** For waitlisted players: (a) report the concrete hard blocker per group (full 11/11, kan inte tiden, differentGroupHard, pinned); (b) generate the priority story directly from data, not from the probe: compare p.priority against min(member.priority) of each full candidate group — 'Grupp C är full och alla placerade har prioritet ≥ din' — optionally backed by a swap probe (insert p, remove lowest-priority unpinned member); (c) if a probe returns BETTER (feasible non-full group), render 'förbättring möjlig — kör om optimeringen' and log it as a solver-quality warning, never as a priority explanation. Add a test asserting no waitlisted player has a feasible non-full candidate after convergence on the golden datasets.
- **[MAJOR]** User-overridable weights can silently destroy the waitlist invariant that the whole §2 design rests on. The doc says every constraint including unassignedPlayer is overridable: (1) disabling unassignedPlayer (enabled=false → ZERO → constraint excluded entirely per verified ConstraintWeightOverrides javadoc) makes waitlisting FREE — the solver will dump players to Kölista to improve levelBalance/groupSizeTarget; (2) reclassifying any wish constraint to MEDIUM (which §9.4/§9.5 of the spec invites and the override mechanism allows) lets a high-weight wish outrank unassignment, breaking both 'soft quality can never justify leaving someone unplaced' and the shed-lowest-priority-first guarantee.
  - **Fix:** Guard-rail in the assembler/validation layer: unassignedPlayer is not disableable, level locked to MEDIUM, weight floor ≥ 1 (422 otherwise); MVP UI restricts field-constraint reclassification to HARD/SOFT only (MEDIUM stays reserved for unassignment, matching the doc's own §3.1 convention); document the reservation in ConstraintMetadata and add a validation test.
- **[MAJOR]** §2.2 PinnedOverflowTest contract row is false as stated: 'Player pinned to a group over max → hard violation stays visible'. With one pinned player in a 12/11 group, the solver simply evicts an unpinned member (to another group at soft cost, or to waitlist at medium cost — both lexicographically cheaper than a hard violation), so hard returns to 0 and the test as specified fails. The implementing agent will burn a session on a wrong expectation.
  - **Fix:** Rewrite the contract: hard violation persists only when the overflow cannot be resolved by moving unpinned entities (e.g. pin all 12 members of a max-11 group). Either construct the test that way, or change the assertion to: pinned player stays put AND the solver restores feasibility by evicting an unpinned player (assert who moved and that hard==0).
- **[MINOR]** Two API-usage corrections for 1.33.0: (1) §9.4 uses SolverJobBuilder.withBestSolutionConsumer / withFinalBestSolutionConsumer — both are @Deprecated in v1.33.0 in favor of the event variants; (2) §9.3 writes 'SolverConfigOverride.withTerminationConfig(...)' as if static — it is an instance method (no static factories exist on the class; §9.4's `new SolverConfigOverride<>().withTerminationConfig(...)` is the correct form).
  - **Fix:** Use withBestSolutionEventConsumer(Consumer<NewBestSolutionEvent<Solution_>>) and withFinalBestSolutionEventConsumer(Consumer<FinalBestSolutionEvent<Solution_>>) — the event exposes solution(). Fix §9.3's phrasing to the instance-method form so the coding agent doesn't reproduce a non-compiling call.
- **[MINOR]** Empty groups are invisible to groupLevelStats-derived constraints: groupBy over forEach(PlayerAssignment) yields no GroupLevelStat for an empty group, so (a) groupOrderByLevel (10.7) cannot detect inversions involving an empty group and the adjacent-pair chain breaks at that link, and (b) coachLevelFit (10.20) is silently zero for a coach on an empty group.
  - **Fix:** Accept and document: empty groups are already penalized by groupMinSize/groupSizeTargetEmpty, which pressures them to fill; alternatively add a complement stream (forEach(Group).ifNotExists(PlayerAssignment...)) that substitutes the group's seeded level band midpoint into the ordering/coach-fit comparison. At minimum, update the metadata text so explanations don't claim ordering was checked for empty groups.
- **[MINOR]** §11.3 rule 4 ('top-N=3 of remaining groups ranked by what-if score delta') requires probing ALL remaining groups to rank them, so a per-player explanation costs ~12 probes (~600 ms at ≤50 ms each), not the ~5 candidates assumed in §11.4's eager-vs-lazy cost analysis (650 analyses ≈ 30 s is really ~1560 ≈ 78 s). Lazy still fits the <1 s M-S3 gate, but barely, and the background-prefetch sizing is off by ~2.4×.
  - **Fix:** Correct §11.4's arithmetic to 12 probes/player; compute all 12 probes in one pass per player and derive rules 1–4 from that single result set (union rule becomes an ordering/labelling rule, not a probe-selection rule); keep the <1 s gate and measure it on the 120-player dataset in CI.
- **[MINOR]** The canonical question fails in one case: if kompisen (Lisa) is herself waitlisted, §11.3 rule 1 yields no candidate group ('every group currently containing any person referenced by p's wishes' is empty), so 'varför hamnade Kalle inte i grupp C med sin kompis' has no FRIEND_WISH candidate and no per-group answer.
  - **Fix:** Add an explicit branch: when a wished person is unassigned, emit the narrative 'Lisa är oplacerad (kölista)' with a link to Lisa's own waitlist explanation, and mark the wish as unevaluable-per-group. §11.5 why-not already covers arbitrary groups as backstop; wire the M-S3 gate (b) fixture to also cover this variant.
- **[MINOR]** Greedy baseline (§9.5) has two nondeterminism gaps: block sort key '(startMinute ASC, courtId ASC)' omits day — two blocks on different days at the same start tie unstably; and coach assignment 'greedily by best coachLevelFit distance then availability' has no specified tie-break when two coaches have equal distance.
  - **Fix:** Sort blocks by (dayOfWeek/epochDay, startMinuteOfDay, courtId, blockId) and break coach ties by (distance ASC, coachProfileId ASC); add the greedy baseline to the cross-platform golden-score CI matrix so drift is build-breaking like the solver's.
- **[MINOR]** CoachSlot synthetic @PlanningId 'groupId * 10 + slotIndex' collides when slotIndex ≥ 10 (id(g,10)==id(g+1,0)); requiredCoachCount is unbounded in the model. Also, @PlanningId javadoc says the member should be a Comparable type, 'usually Long or String' — primitive long works via boxing but deviates from the documented convention.
  - **Fix:** Use groupId * 100 + slotIndex and validate requiredCoachCount ≤ 99 in the assembler (or give CoachSlot rows their own DB sequence); switch @PlanningId fields to Long across the three entities to match the javadoc convention and remove any doubt.
- **[MINOR]** The empty-group complement constraints introduce extra constraint keys (groupSizeTargetEmpty; the 10.5 complement is unnamed) that must fan out from the SAME DB ConstraintWeightConfig row as their parent, or a user weight change will apply to non-empty groups only — a silent inconsistency the key-validation test won't catch since both keys are individually valid.
  - **Fix:** Name both complements explicitly (groupSizeTargetEmpty, groupMinSizeEmpty), make the override-mapping layer fan one DB row out to the parent+complement key pair, and add a test asserting the fan-out (change weight → both constraints' applied weight changes in ScoreAnalysis).
- **[MINOR]** §15.5 class-level pinning ('optimera endast spelare' pins all GroupSchedules at current values) is impossible on a plan that has never been solved: GroupSchedule.trainingBlock is null and NOT unassignable, so the pre-solve validator correctly 422s — but the design never states this ordering constraint, and the UI will expose the checkbox before a first solve. Related: 10.2's 'satisfied by construction' claim silently depends on groupCount ≤ activeBlockCount surviving manual group additions and block deactivations after generation.
  - **Fix:** Document and enforce: 'optimera endast X' modes require a fully assigned plan (validator message tells the user to run a full solve first); add CRUD-time validation that deactivating a block or adding a group keeps groupCount ≤ activeBlockCount, or downgrade the 10.2 metadata text from 'garanteras av modellen' to 'överträdelser fångas av trainingBlockCapacity'.
- **[MINOR]** Milestone realism: M-S1 bundles the entire domain model, assembler, GroupGenerator, LevelService, solve loop + 3 REST endpoints, GreedyBaselineService, ~12 hard constraints with justifications and verifier tests, two arch/guard tests, AND standing up the macOS+Windows CI determinism matrix — realistically 1.5–2 agent sessions. The milestones also silently assume Flyway schema for OptimizationRun, ConstraintWeightConfig and ExplanationRecord exists (persistRun is in M-S1 scope) without listing it anywhere.
  - **Fix:** Move GreedyBaselineService and LevelService to M-S2 (nothing in M-S1's gates needs them; greedy is only needed once soft constraints exist for comparison), or split M-S1 into M-S1a (domain+constraints+ConstraintVerifier gates) and M-S1b (assembler+solve loop+CI matrix). Add an explicit prerequisite line naming the Flyway migrations each milestone consumes.

### API checks

- @PlanningVariable(allowsUnassigned = true): CONFIRMED — v1.33.0 PlanningVariable.java has allowsUnassigned() default false; nullable() is @Deprecated(forRemoval, since 1.8.0); javadoc confirms unassigned adds null to the value range (so CH does evaluate null, supporting the waitlist design).
- @PlanningPin: CONFIRMED — v1.33.0 PlanningPin.java javadoc: 'applies to all the planning variables of that planning entity'; @Target({METHOD, FIELD}) boolean member.
- ConstraintWeightOverrides: CONFIRMED — interface with static none() and of(Map<String,Score_>), instance getConstraintWeight(String)/getKnownConstraintNames(); detected as a member on the @PlanningSolution class; javadoc: zero weight 'will be excluded from processing, and the solver will behave as if it did not exist in the ConstraintProvider' (exactly as the design's disable semantics claim).
- justifyWith/indictWith: CONFIRMED — UniConstraintBuilder.justifyWith(BiFunction<A, Score_, ConstraintJustification_>) and indictWith(Function<A, Collection<Object>>) in v1.33.0.
- SolutionManager: CONFIRMED — analyze(Solution_), analyze(Solution_, ScoreAnalysisFetchPolicy[, SolutionUpdatePolicy]), update(Solution_[, SolutionUpdatePolicy]), explain(...), recommendAssignment(Solution_, EntityOrElement_, Function[, ScoreAnalysisFetchPolicy]) all present in Community core at v1.33.0; recommendFit @Deprecated(forRemoval, since 1.15.0).
- ScoreAnalysisFetchPolicy: CONFIRMED — exactly FETCH_ALL, FETCH_SHALLOW, FETCH_MATCH_COUNT in v1.33.0.
- ScoreAnalysis: CONFIRMED — record(score, constraintMap, isSolutionInitialized); diff(other) documented 'in the direction of this - other' (design §12.2 direction correct); diff throws if only one side carries MatchAnalysis and relies on ConstraintJustification equality (design's id-carrying records satisfy this); summarize()/summarize(int) exist and require FETCH_ALL.
- ConstraintAnalysis / MatchAnalysis: CONFIRMED — ConstraintAnalysis(constraintRef, weight, score, matches, matchCount) exactly as claimed; MatchAnalysis(constraintRef, score, justification) with justification() returning ConstraintJustification.
- TerminationConfig: CONFIRMED — spentLimit, unimprovedSpentLimit (Duration), stepCountLimit, unimprovedStepCountLimit (Integer), moveCountLimit, scoreCalculationCountLimit (Long) all exist with with* builders in v1.33.0.
- EnvironmentMode: CONFIRMED — NO_ASSERT (REPRODUCIBLE @Deprecated forRemoval since 1.20.0, 'Prefer NO_ASSERT'), PHASE_ASSERT is the documented default, STEP_ASSERT (FAST_ASSERT deprecated 1.20.0), TRACKED_FULL_ASSERT and NON_REPRODUCIBLE exist. Design's §9.2 table is exactly right.
- SolverJobBuilder: PARTIALLY WRONG — withProblemId/withProblem/withExceptionHandler(BiConsumer)/withConfigOverride(SolverConfigOverride)/run() CONFIRMED, but withBestSolutionConsumer and withFinalBestSolutionConsumer are @Deprecated in v1.33.0; the correct 1.33 API is withBestSolutionEventConsumer(Consumer<NewBestSolutionEvent>) / withFinalBestSolutionEventConsumer(Consumer<FinalBestSolutionEvent>) with event.solution().
- SolverConfigOverride: PARTIALLY WRONG as notated — withTerminationConfig is an INSTANCE method (plus withTerminationSpentLimit/withTerminationUnimprovedSpentLimit); no static factory, so §9.3's 'SolverConfigOverride.withTerminationConfig(...)' does not compile as written; §9.4's new SolverConfigOverride<>() form is correct.
- forEach semantics: CONFIRMED — v1.33.0 ConstraintFactory javadoc: forEach filters to entities where each genuine variable is non-null; forEachIncludingUnassigned removes that filtering; forEachIncludingNullVars deprecated. The design's null-friend/null-coach exclusion reasoning in 10.11/10.15 is valid.
- ConstraintVerifier test API: CONFIRMED — ConstraintVerifier.build(provider, solutionClass, entityClasses...), verifyThat(BiFunction), SingleConstraintAssertion.penalizes(long)/penalizesBy(matchWeightTotal — 'sum of match weights', matching the design's test expectations)/rewards/justifiesWith(ConstraintJustification...)/indictsWith in timefold-solver-test v1.33.0.
- ConstraintCollectors.loadBalance: CONFIRMED available in Community core; LoadBalance.unfairness() returns BigDecimal (six decimal places) — the design's rejection rationale is accurate.
- Spring starter: CONFIRMED — TimefoldProperties prefix 'timefold', DEFAULT_SOLVER_CONFIG_URL 'solverConfig.xml' picked up from classpath, solverConfigXml property; starter autoconfigures SolverManager/SolverFactory/SolutionManager beans.
- Default local search: CONFIRMED — DefaultLocalSearchPhaseFactory defaults to LocalSearchType.LATE_ACCEPTANCE via Objects.requireNonNullElse in both buildAcceptor and buildForager.
- Enterprise-only list: CONFIRMED — v1.33.0 enterprise-edition.adoc lists nearby selection, multithreaded incremental solving, partitioned search, automatic node sharing, throttling best-solution events, multistage moves (all as the design claims) plus constraint profiling (unlisted in the design but unused, harmless).
- HardMediumSoftLongScore: CONFIRMED — of(long,long,long), ofHard/ofMedium/ofSoft, ZERO, hardScore()/mediumScore()/softScore(), isFeasible() == hardScore ≥ 0.
- @PlanningId: CONFIRMED with nit — javadoc requires a Comparable type, 'usually Long or String'; primitive long works via boxing but Long is the documented convention (see minor finding).
- GroupOrderByLevel expressibility: CONFIRMED expressible in Constraint Streams — groupBy(key, sumLong, count) → TriConstraintStream → map(TriFunction) → UniConstraintStream, then UniConstraintStream.join(otherUniStream, Joiners.equal(leftFn, rightFn)) with cross-multiplied integer filter; all operators exist in 1.33 and the 201-point test arithmetic checks out. Caveat: blind to empty groups (see minor finding).
