# Adversarial Review Round 1 (feasibility + spec coverage)


## Feasibility red-team

### Verdict

The architecture document is unusually well-researched — the Timefold 1.x pin rationale, the jlink-as-resource decision, and the browser-first dev workflow all verify as sound — but the combined plan has two blocking defects. First, the two documents contradict each other on ORM, Java packaging mechanism, UI kit, Vite major, repo root (the product doc's layout puts the confidential xlsx inside the git repo), and milestone ordering; agents fed both will produce an incoherent codebase. Second, the centerpiece determinism gate asserts exact golden-score equality across mac/win CI runners while every solver profile uses wall-clock termination — Timefold's own docs say time-based termination is not reproducible across differing hardware, so the gate fails by construction and must switch to step/move-count termination for tests. Beyond those, the macOS ad-hoc signing step runs after the .dmg is built and --deep won't sign the JRE binaries in Resources, CI never launches the actual bundled artifact, the solver design section is missing while carrying the hardest unresolved problems (integer score scaling, per-variable lock semantics, cross-plan blocking), and the plan overlooks that Timefold Plus is free for nonprofits — directly relevant to this customer. All are fixable with concrete, mostly small changes, but reconciliation and the solver design rerun must happen before any agent implementation session starts.

### Problems

- **[BLOCKER]** The two design documents contradict each other on core technical decisions, and coding agents will be fed both. Conflicts: persistence (JdbcClient + Flyway, no JPA vs Spring Data JPA + Hibernate community SQLiteDialect with ddl-auto=validate — the latter is experimental-tier and notorious for type-affinity validation mismatches against Flyway-created SQLite schemas); Java packaging (jlink runtime directory as Tauri resource vs 'jpackage --type app-image bundled as Tauri resource' — a jpackage app-image on macOS is a nested .app inside the outer .app's Resources, which breaks signing/Gatekeeper and adds a redundant native launcher); UI kit (Mantine + AG Grid vs Ant Design 5); Vite 7 vs Vite 8; auth header (X-GP-Token vs X-App-Token); package namespace (se.klubb.groupplanner vs se.gruppindelning); milestone order (packaging in Milestone 0 vs packaging last in M9). Worst: the product doc's repo layout makes /Users/eskilnord/beach-gruppindelning/ itself the repo root — the directory that contains the confidential 'Torsdagsträning VT26.xlsx' — directly contradicting the architecture doc's app/-subdirectory confidentiality firewall.
  - **Fix:** Reconcile before any agent session starts, and record winners as ADRs. Adopt the architecture doc's positions on every conflict: JdbcClient + Flyway (no Hibernate/JPA on SQLite), jlink resource directory (kill jpackage), git repo rooted at app/ so the confidential xlsx is physically outside the tree, packaging-first walking skeleton (M0), one UI kit, one token header name. Delete or annotate the losing alternatives in the merged plan so agents never see two answers.
- **[BLOCKER]** The cross-platform determinism gate is broken as designed. All solver profiles use wall-clock termination (Snabb 10s / Normal 60s / Grundlig 120s), and SolverRegressionTest asserts exact equality of score + hard-violation count + assignment hash against a golden file on both macos-latest and windows-latest runners. Timefold's own docs state that reproducible mode does not guarantee identical results across runs with time-spent termination, because available CPU time differs between machines — GitHub mac and windows runners differ substantially in speed, and even the same runner varies run to run. The gate will flake or hard-fail by construction, and the M6 acceptance criterion 'same score twice with seed 0' fails locally too under time termination. The spec's own §22.3/§16.5 goal is only achievable with deterministic termination.
  - **Fix:** Regression and CI tests must use deterministic termination: step-count, move-count, or unimproved-step-count limits (never time-based), with a fixed seed and sorted input — then exact golden-score equality across OSes is actually achievable. Keep wall-clock termination only for interactive user runs, and document in the plan (and UI copy) that interactive runs are reproducible only in the sense of §22.3's fallback (identical only under identical compute). Also avoid Simulated Annealing (time-gradient-dependent); prefer Late Acceptance.
- **[MAJOR]** The Timefold licensing analysis is incomplete and the whole pinned stack is on an end-of-life track. Verified: 2.x did move ScoreAnalysis/recommendation APIs to paid 'Plus', and 1.33.0 is the last 1.x feature release with only quarterly bugfixes through 2026 — but Timefold simultaneously announced that Enterprise (including Plus) is FREE for nonprofits, and this customer is a Swedish nonprofit club. Neither document mentions this. Combined with Spring Boot 3.5 OSS support having ended 2026-06-30, the app ships on a framework and a solver line that are both effectively EOL at delivery, with the stated post-MVP escape hatches unexamined.
  - **Fix:** Keep the 1.33.0 pin for MVP (no license administration, Java 17/21 freedom, stable docs for agents) but verify and record the Timefold-for-nonprofits Plus terms now, and write the ADR upgrade path as: MVP on 1.33 → nonprofit Plus license on 2.x (Java 21 already bundled, so the runtime is ready). Confirm the free-nonprofit terms cover this use before the explainability layer hard-codes 1.x-only API shapes; keep the explain/what-if code behind a thin internal interface to ease the 2.x migration.
- **[MAJOR]** The macOS ad-hoc signing pipeline is ordered wrong and Gatekeeper friction is understated. package.sh runs 'codesign --force --deep --sign -' AFTER 'tauri build' — but tauri build has already produced the .dmg, so the shipped disk image contains the app as it was before the extra signing step. Worse, --deep is deprecated and only descends into nested code in standard locations (Frameworks/, PlugIns/); Mach-O executables and dylibs sitting under Contents/Resources/jre/ (bin/java, lib/jspawnhelper, dozens of .dylibs) are treated as sealed resources, not signed code — on Apple Silicon an unsigned or seal-broken arm64 binary is killed outright. jlink output signing state also varies by JDK build. Separately, a quarantined ad-hoc-signed app on macOS 15 frequently produces the '"appen" är skadad' (damaged) dialog rather than the documented 'Open Anyway' path, which a kansli user cannot recover from without Terminal.
  - **Fix:** Sign the jlink runtime binaries individually (find resources/jre -type f -perm +111 plus all .dylib/.jnilib, codesign --force --sign - each) immediately after build-jre.sh, BEFORE tauri build, and let Tauri's signingIdentity '-' seal the outer bundle and then build the dmg — delete the post-build --deep step entirely. Make 'download the .dmg on a clean quarantined Mac and open it following only docs/installation-mac.md' an explicit Milestone-0 exit criterion; if the 'damaged' dialog appears, escalate immediately to Apple Developer Program + notarization (use Apple's nonprofit fee waiver) instead of shipping xattr instructions.
- **[MAJOR]** CI never exercises the actual bundled artifact. The desktop-smoke job runs steps 1–3 of package.sh and launches resources/jre/bin/java from the pre-bundle build tree — the Tauri bundling step (resource copy into .app/Contents/Resources and NSIS packaging), the installed resource-dir path resolution, and the shell's spawn/handshake path are never executed in CI. This is exactly where known failure classes live: Tauri resource bundling stripping the executable bit (PermissionDenied on spawn, cf. tauri-apps/tauri#4653), per-platform resource path differences, and the signing-seal issues above. The first time the real artifact runs is on a human's machine.
  - **Fix:** Add a release-workflow (and nightly) job per OS that takes the built artifact, installs it for real (hdiutil attach + copy .app on macOS; NSIS silent install /S on Windows), launches the installed app headless or via a --smoke flag that runs the spawn/handshake and exits with the health-check status, and asserts success. Defensively, in backend.rs, before spawning, restore 0o755 on jre/bin/* and lib/jspawnhelper on Unix — it is cheap insurance against permission stripping.
- **[MAJOR]** The solver design section is missing entirely ('solver agent failed'), and the fragments scattered in the other two docs leave the hardest problems undesigned: (1) integer score discipline — the schema stores levels as REAL and LevelBalance/nivåspridning implies variance/spread math, but the architecture doc mandates integer HardSoftLongScore for determinism; the scaling scheme (e.g. levels ×100, spread as integer sum of absolute deviations, loadBalance collector usage) is nowhere specified; (2) lock granularity — Timefold 1.x @PlanningPin is entity-level, so pinning a Group entity pins BOTH its trainingBlock and coach variables, while spec §15.2/§15.3 require locking tid/bana and tränare independently; the 'LockedAssignmentHard + pinned entities' phrase doesn't resolve this; (3) how group count and level ranges are determined pre-solve; (4) how CoachCannotTrainAndCoachSameTime and SavedPlanResourceBlock work across ActivityPlans (cross-plan facts in the solution). M6 is sized as one agent session on top of this void.
  - **Fix:** Rerun the solver design before finalizing the merged plan. It must specify: integer scaling for all level math (ban doubles in constraint code, enforced by a test), lock mechanics per variable (hard constraint MatchesLock facts for group-block and group-coach locks; @PlanningPin only for PlayerAssignment where it maps 1:1), the pre-solve group-generation algorithm, the cross-plan blocking fact model, and the deterministic-termination test configs from the blocker above. Re-estimate M6 afterward — expect 2–3 sessions, not one.
- **[MINOR]** The pinned jlink module list omits java.logging. Embedded Tomcat (JULI) requires java.util.logging, so the backend will fail at startup in the trimmed runtime; the pinned list (java.base,java.sql,java.naming,java.desktop,java.management,java.instrument,java.net.http,java.security.jgss,java.xml,jdk.unsupported,jdk.crypto.ec) does not include it. The smoke test would catch it late; also verify jdk.crypto.ec against the exact JDK 21 jmods (it was folded into java.base in later JDKs) — for a plain-HTTP localhost backend it may be droppable entirely.
  - **Fix:** Add java.logging (and re-run jdeps against the actual fat jar rather than trusting the hand-pinned list; diff jdeps output vs the pinned list in CI so drift fails the build).
- **[MINOR]** Stale/conflicting Timefold environment-mode naming. EnvironmentMode.REPRODUCIBLE has been deprecated since 1.19 (removed in 2.0); in 1.33 the default is PHASE_ASSERT (reproducible + cheap assertions) and the old REPRODUCIBLE maps to NO_ASSERT. The architecture doc's 'environmentMode=REPRODUCIBLE' property will emit deprecation warnings and confuse agents cross-referencing 1.33 docs; the product doc's parenthetical is closer but conflates it as '(PHASE_ASSERT)' under the reproducible name.
  - **Fix:** Standardize explicitly: NO_ASSERT for user-facing production solves (fastest, still reproducible), PHASE_ASSERT or STEP_ASSERT in tests/CI. Record the exact enum names in CLAUDE.md alongside the 1.33 docs-URL rule.
- **[MINOR]** The import pipeline throws away numeric cell values: keeping every cell only as a DataFormatter string with Swedish locale means rankingPoints arrive as '1 234,5' (comma decimal, possibly non-breaking-space thousands separator) and must be re-parsed locale-aware — a classic silent-corruption spot for mixed-type columns (some cells NUMERIC, some typed as text). Also the CSV fallback charset should be windows-1252 rather than ISO-8859-1 (identical for å/ä/ö but not for typographic characters common in comment fields, e.g. ’ … –).
  - **Fix:** For NUMERIC/DATE cells keep the raw double / java.time value alongside the formatted string and map numeric targets from the typed value, using locale-aware parsing only for text-typed cells; fall back to windows-1252, and add fixture rows covering '1 234,5', text-formatted numbers, and cp1252-only characters.
- **[MINOR]** Small contract gaps between the docs' handshake and API surface: the shutdown endpoint (POST /api/system/shutdown) required by the shell's exit sequence is absent from the product doc's REST surface; and the token filter as described rejects all non-token requests while the shell's readiness poll and the OpenAPI typegen step (/v3/api-docs fetched by openapi-typescript in dev) both need defined token behavior.
  - **Fix:** Add /api/system/shutdown to the API list; state explicitly that the shell's health poll sends the token, and that in the dev profile the fixed token also guards /v3/api-docs (or exempt it in dev only). Trivial, but agents implement exactly what the contract says.

## Spec coverage

### Verdict

The architecture document is strong and its load-bearing claim checks out (Timefold 2.x really did move ScoreAnalysis to the paid Plus edition, so the 1.33 pin is correct and essential), and the product document covers most of §23 competently — import wizard, capacity fields, comment-off-by-default export, and person matching all trace cleanly to the spec. However, the combined design has a hole exactly where the risk is: the solver design is missing entirely, and with it the answers to pinning semantics, the hard constraints' implementations, and — critically — what happens when demand exceeds capacity, even though the real file's Kölista makes overcapacity the expected case, not an edge case. The explainability design as written cannot reliably answer the spec's one canonical question (why not the friend's group) because alternatives are limited to top-3 by score. Finally, the two documents contradict each other on repo layout (with confidentiality implications), JRE packaging, frontend stack, and milestone ordering — the product plan defers all packaging and cross-platform determinism to the final milestone, which the architecture document itself correctly argues is the wrong order. These must be reconciled and the solver design produced before implementation begins.

### Problems

- **[BLOCKER]** The solver design is missing entirely ('solver agent failed'), leaving the spec's riskiest and most central component with only a one-paragraph sketch in the product doc. Undesigned: the planning model details (nullable group variable or not), how LockedAssignmentHard (§10.23) relates to Timefold pinning (@PlanningPin vs a hard constraint — the product doc lists BOTH 'LockedAssignmentHard + pinned entities', which is redundant and can distort scores/explanations), the Constraint Streams implementation of the hard ones like GroupOrderByLevel (§10.7, group-aggregate comparisons are genuinely tricky), CoachCannotTrainAndCoachSameTime (§10.17) joining a person's coach_profile and participant_profile within one solve, LevelBalance spread math and integer scaling, and how group count/level ranges are derived in POST /groups/generate. M6 is acknowledged as the largest milestone and it is the only one with no design behind it.
  - **Fix:** Re-run the solver design before any implementation of M6. Require it to specify: exact planning entities/variables and nullability, pinning mechanism (use @PlanningPin for locks; drop the redundant hard constraint or use it only for cross-plan blocks), a per-constraint implementation sketch for all of §10.1–10.24 with the score level and scaling for each, the group-generation algorithm (count = f(participants, targetSize, active blocks)), and the ConstraintVerifier test matrix (§26.13 demands a test per hard constraint).
- **[BLOCKER]** Overcapacity/waitlist behavior is undefined, yet the real dataset has a Kölista section and §12.4's own example (122 anmälda vs 110 target capacity) shows demand pressure is the normal case. The schema has participant_profile.waitlisted and the export has a Kölista section, but nothing says who decides who is unplaced: if the solver's group variable is non-nullable and participants > maxCapacity, every solve ends with unavoidable GroupMaxSizeHard violations and an arbitrary distribution of the overflow; if it is nullable, there is no designed constraint (e.g. priority-weighted unassignment penalty using the §9.2 Prioritet field) nor any explanation story for 'why was I waitlisted'. The capacity view warns about waitlist risk but the product has no path through it.
  - **Fix:** Pick and design one mechanism: (a) pre-solve gate — solve is blocked until kansliet manually marks enough participants waitlisted that count ≤ maxCapacity (simplest, keeps control with kansliet per §2.4), or (b) nullable group variable with a medium/high-soft 'unassigned' penalty scaled by the Prioritet field, waitlisted players surfaced in Resultatvyn with their own explanation record. Whichever is chosen, add acceptance tests where participants > maxCapacity and assert deterministic, explainable waitlist output feeding the export's Kölista section.
- **[MAJOR]** The two design documents contradict each other on decisions coding agents will implement from: (1) repo location — architecture doc mandates the git repo in app/ so the confidential xlsx is physically outside the tree, while the product doc's repo layout puts backend/, frontend/, testdata/ at /Users/eskilnord/beach-gruppindelning/ root, directly next to Torsdagsträning VT26.xlsx; (2) JRE packaging — arch explicitly rejects jpackage in favor of a jlink resource dir, product doc specifies 'jpackage --type app-image'; (3) frontend stack — Mantine 8 + AG Grid + Vite 7 + Zustand vs Ant Design 5 + Vite 8; (4) Windows artifact — NSIS .exe primary vs .msi; (5) milestone ordering — arch requires an M0 walking skeleton with packaging + mac/win CI proven first, product plan defers all packaging and cross-platform regression to M9, the last milestone, so §22.3/§23.10 (the hardest cross-cutting risk: determinism, bundled JRE, unsigned-app friction, Electron-fallback gate) is only exercised at the very end.
  - **Fix:** Reconcile into a single ADR set before implementation starts. Adopt the architecture doc's positions (app/ subdir firewall, jlink resources, NSIS primary, one UI stack — pick Mantine or AntD once) and restructure the milestone plan: insert M0 = walking skeleton (packaged shell + handshake + mac/win CI with the golden-score gate), and run the cross-platform solver regression from the first milestone that has a solver (M6), not at M9.
- **[MAJOR]** The spec's canonical question — 'Varför hamnade Kalle i grupp Y och inte i grupp C med sin kompis?' (§27, §17.2, §23.7) — is not guaranteed answerable: alternative-group comparisons are computed only 'for top-3 candidate groups' by score. The group containing the friend is frequently NOT among the top-3 alternatives (that is precisely why the wish was broken), so explanation_record.alternative_groups_json will lack the one comparison the user actually asks for. §23.7 also requires 'varför minst en alternativ grupp inte valdes' with reasons like 'Grupp C var full' — those reasons must come from the friend's group specifically.
  - **Fix:** Define the candidate set as the union of: every group containing a person referenced by the participant's personRelation wishes (fulfilled or broken), every coach-wish group, the previous group (continuity), plus top-N by score-diff. Additionally add an on-demand 'Varför inte grupp X?' selector in the explain drawer that runs the what-if analyze diff for any group the user picks — this also bounds the post-solve cost (130 participants × eager what-if analyses per run should be measured; prefer lazy computation with caching over eager persistence for all participants).
- **[MAJOR]** Cross-plan conflict handling only exists via the locked-plan mechanism (saved_plan_resource_usage materialized when status→locked). But §13.2 says the system 'måste förhindra överlapp' for a coach who also plays (the Anna Herr/Dam example is cross-plan), §19.2 requires Säsongsvyn to show 'Konflikter', and §14.4's option 'Visa konflikter men tillåt ändå' presupposes conflict computation without blocking — none of which is designed. Two draft plans optimized independently can double-book the same person/coach/court-time with no detection anywhere. Also §14.4 speaks of 'tidigare sparade planer' while the design blocks only locked ones, and matching resource usage on day_of_week+times breaks for date-based TimeSlots (Sommarläger).
  - **Fix:** Add a season-level ConflictService: compute person/coach/court×time overlaps across ALL activity plans in a season regardless of status, surface them in Säsongsvyn, Resultatvyn warnings, and Planeringskartan. Materialize resource usage on save (not only lock) and let §14.4's checkboxes decide whether usage acts as hard block, soft warning, or 'visa konflikter men tillåt ändå' at solve time. Define overlap matching on concrete date+time when date is set, day_of_week+time otherwise.
- **[MAJOR]** §21.2 requires comments to be deletable/anonymizable ('ska kunna raderas/anonymiseras') — no endpoint, UI, or milestone covers this. Worse, optimization_run.input_snapshot_json and saved_plan.snapshot_json are wholesale snapshots; if they serialize participant profiles as-is, importedComment/internalNote get copied into immutable JSON blobs that a later delete of participant_profile.imported_comment will not reach, silently defeating the deletion requirement.
  - **Fix:** Add 'Radera/anonymisera kommentarer' actions (per participant and per plan) to the design and a milestone (fits M4 or M8). Specify the snapshot schemas explicitly to EXCLUDE importedComment and internalNote (the solver never needs them per §2.2, so snapshots don't either), and add an automated test asserting no comment text appears in any *_json column or in solver input after a full import→solve→save cycle.
- **[MINOR]** The import validation list omits 'ogiltiga tider' (invalid times), one of the eight explicitly required flags in §8.6 — the design lists the other seven. Since time columns are imported as reference text, no time validation happens anywhere in the wizard.
  - **Fix:** Add validation for any column mapped to a time-typed target (timeRelation custom fields), and flag unparseable time-looking values in reference columns as warnings ('ogiltig tid — kontrollera manuellt') so the §8.6 list is fully covered; add it to M3's acceptance checks.
- **[MINOR]** Determinism configuration is confused in the product doc: 'environmentMode reproducible default (PHASE_ASSERT)' conflates two different modes — PHASE_ASSERT is a slow assertion/debug mode that re-scores from scratch, not the production reproducible mode; shipping it would multiply solve times. The two docs also disagree on score type: HardSoftScore (product) vs integer-scaled HardSoftLongScore (arch), and only the latter addresses §4.5.7 decimal normalization.
  - **Fix:** Production config: environmentMode=REPRODUCIBLE + fixed randomSeed; use FULL_ASSERT/PHASE_ASSERT only in tests. Standardize on HardSoftLongScore with documented integer scaling for level values, per the architecture doc.
- **[MINOR]** Several smaller spec/product gaps: (a) §13.1 allows importing coaches ('manuellt eller importera dem') but the wizard only maps participant targets; (b) explanation_records are keyed to optimization_run, so after a manual apply-move the explain drawer shows stale justifications with no staleness indicator; (c) §23.1's literal acceptance criterion is that the REAL example file imports, but every planned test uses synthetic fixtures and no milestone schedules a manual verification pass with the confidential file (which can never enter CI).
  - **Fix:** (a) Add a coach mapping target or a 'skapa tränare från person' bulk action post-import; (b) mark explanations 'Inaktuell — baserad på körning N' after any manual move/lock and invalidate the what-if baseline to current DB state; (c) add an explicit manual acceptance checklist item at M3 (import) and M8 (export layout vs the council's current file) executed by the user on the local machine with the real file.
